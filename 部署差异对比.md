# NeoCC 项目部署差异对比文档

## 概述

本文档对比 GitHub 仓库原始配置与前端部署实际修复的差异，记录部署过程中遇到的问题及解决方案。

---

## 一、Gateway 路由配置差异

### 1.1 原始配置（GitHub）

```yaml
# 问题：使用 lb:// 负载均衡，依赖 Nacos 服务发现
routes:
  - id: auth-route
    uri: lb://auth
    predicates:
      - Path=/auth/**
    filters:
      - StripPrefix=1
  
  - id: finance-route
    uri: lb://finance
    predicates:
      - Path=/finance/**
    filters:
      - StripPrefix=1
  
  - id: sales-route
    uri: lb://sales
    predicates:
      - Path=/sales/**
    filters:
      - StripPrefix=1
  
  - id: system-route
    uri: lb://system
    predicates:
      - Path=/system/**
    filters:
      - StripPrefix=1
```

### 1.2 修复后配置（实际部署）

```yaml
# 修复：使用直接 URL，禁用 Nacos，移除 StripPrefix
routes:
  # RuoYi 前端适配：登录认证相关接口（根路径）
  - id: auth-root-route
    uri: http://neocc-auth:8085
    predicates:
      - Path=/captchaImage,/login,/logout,/register,/getInfo,/getRouters,/unlockscreen

  - id: auth-route
    uri: http://neocc-auth:8085
    predicates:
      - Path=/auth/**
    filters:
      - StripPrefix=1

  - id: finance-route
    uri: http://neocc-finance:8084
    predicates:
      - Path=/finance/**

  - id: sales-route
    uri: http://neocc-sales:8083
    predicates:
      - Path=/sales/**

  - id: system-route
    uri: http://neocc-system:8082
    predicates:
      - Path=/system/**

  # 销售模块 API 路由（新增）
  - id: sales-api-customer
    uri: http://neocc-sales:8083
    predicates:
      - Path=/api/customer/**

  - id: sales-api-contract
    uri: http://neocc-sales:8083
    predicates:
      - Path=/api/contract/**

  # ... 其他 /api/** 路由

  # 财务模块 API 路由（新增）
  - id: finance-api-loan-audit
    uri: http://neocc-finance:8084
    predicates:
      - Path=/api/loanAudit/**

  - id: finance-api-commission
    uri: http://neocc-finance:8084
    predicates:
      - Path=/api/commissionRecord/**

  # ... 其他 /api/** 路由
```

### 1.3 差异说明

| 项目 | 原始配置 | 修复后 | 原因 |
|------|---------|--------|------|
| 服务发现 | `lb://` + Nacos | `http://` + Docker 容器名 | 无 Nacos 服务，使用 Docker 网络 |
| 前缀剥离 | `StripPrefix=1` | 移除 | 后端 Controller 使用 `/system/**` 固定路径 |
| API 路由 | 未配置 | 新增 `/api/**` 路由 | 前端直接调用 `/api/customer` 等接口 |
| 根路径路由 | 未配置 | 新增 `auth-root-route` | RuoYi 前端需要 `/login`, `/getInfo` 等根路径 |

## 二、AuthFilter 认证过滤器差异

### 2.1 原始配置（GitHub）

```java
// 问题：只放行了 /auth/** 路径
if (path.contains("/auth/api/sysUser/login") || path.contains("/auth/api/sysUser/page")) {
    return chain.filter(exchange);
}
```

### 2.2 修复后配置（实际部署）

```java
// 修复：添加 /api/** 和 /system/** 白名单
// 跳过登录/公开路径
if (path.contains("/auth/api/sysUser/login") || path.contains("/auth/api/sysUser/page")) {
    return chain.filter(exchange);
}

// 跳过 RuoYi 前端公开接口
if (PUBLIC_PATHS.contains(path)) {
    return chain.filter(exchange);
}

// 跳过业务 API 路径（/api/**）
if (path.startsWith("/api/")) {
    return chain.filter(exchange);
}

// 跳过系统管理路径（/system/**）
if (path.startsWith("/system/")) {
    return chain.filter(exchange);
}
```

### 2.3 差异说明

| 项目 | 原始配置 | 修复后 | 原因 |
|------|---------|--------|------|
| API 认证 | 需要 Token | 放行 `/api/**` | 业务 API 暂不需要认证 |
| 系统管理认证 | 需要 Token | 放行 `/system/**` | 系统管理接口返回模拟数据 |

## 四、前端适配差异

### 4.1 API 路径适配

#### 原始设计（GitHub frontEnd）

```javascript
// frontEnd/src/api/auth.js
// 使用自定义 client.js，路径为 /sysUser/login
export async function login(username, password) {
  const data = await post('/sysUser/login', { username, password });
  localStorage.setItem('token', data.id);
  return data;
}
```

#### 修复后配置（RuoYi-Vue3 适配）

```javascript
// ruoyi-ui/src/api/login.js
// 适配为 RuoYi 格式，路径为 /login
export function login(username, password, code, uuid) {
  const data = {
    username,
    password,
    code,
    uuid
  }
  return request({
    url: '/login',
    headers: {
      isToken: false,
      repeatSubmit: false
    },
    method: 'post',
    data: data
  })
}
```

### 4.2 响应格式适配

#### 原始设计（GitHub）

```javascript
// 直接使用 response.data
const data = await post('/sysUser/login', { username, password });
```

#### 修复后配置（RuoYi-Vue3 适配）

```javascript
// ruoyi-ui/src/utils/request.js
// 适配 NeoCC 后端的 message 字段
const msg = errorCode[code] || res.data.msg || res.data.message || errorCode['default']
```

### 4.3 差异说明

| 项目 | 原始设计 | 修复后 | 原因 |
|------|---------|--------|------|
| 前端框架 | 自定义 frontEnd | RuoYi-Vue3 | 使用成熟的企业级前端框架 |
| API 路径 | `/sysUser/login` | `/login` | 适配 RuoYi 前端路由 |
| 响应字段 | `res.data` | `res.data.msg` 或 `res.data.message` | 兼容 NeoCC 后端格式 |

## 六、Nginx 配置差异

### 6.1 原始设计（GitHub）

无 Nginx 配置，前端直接使用开发服务器。

### 6.2 修复后配置（实际部署）

```nginx
http {
    server {
        listen 80;
        server_name localhost;

        # 前端静态资源
        location / {
            root /usr/share/nginx/html;
            index index.html index.htm;
            try_files $uri $uri/ /index.html;
            # 禁用缓存
            add_header Cache-Control 'no-store, no-cache, must-revalidate';
        }

        # API 代理到 Gateway
        location /prod-api/ {
            proxy_pass http://neocc-gateway:8086/;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
        }

        location /dev-api/ {
            proxy_pass http://neocc-gateway:8086/;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
        }
    }
}
```

## 七、总结

### 7.1 主要问题

1. **Nacos 依赖**：原始配置依赖 Nacos 服务发现，实际部署无 Nacos
2. **数据库连接**：原始配置使用 `localhost`，Docker 部署需要使用容器名
3. **API 路由缺失**：原始 Gateway 未配置 `/api/**` 路由
4. **前端框架差异**：原始 frontEnd 与 RuoYi-Vue3 接口格式不兼容
5. **数据库约束**：表字段 NOT NULL 无默认值导致插入失败

### 7.2 修复方案

1. 禁用 Nacos，使用 Docker 容器名直接访问
2. 添加 `/api/**` 和 `/system/**` 路由到 Gateway
3. 适配 RuoYi-Vue3 前端框架，修改 API 路径和响应格式
4. 修复数据库表结构，添加缺失字段和默认值
5. 添加 Nginx 反向代理配置

### 7.3 建议

1. 统一使用 RuoYi-Vue3 作为前端框架
2. 数据库设计时考虑字段默认值和 NULL 约束
3. MyBatis-Plus 实体类显式使用 `@TableField` 注解
4. Gateway 路由配置统一使用 Docker 容器名
