# 大富翁贷款管理系统 — 核心技术文档

---

## 四、核心技术

本系统采用前后端分离的微服务架构，后端基于 Spring Cloud 微服务生态构建，前端基于 RuoYi-Vue 框架进行改造开发。

### 4.1 后端

#### 4.1.1 基础框架：Spring Boot 3.2.5

Spring Boot 是由 Pivotal 团队（现隶属于 VMware）开发的开源 Java 框架，旨在简化 Spring 应用的初始搭建与开发过程。它通过"约定优于配置"的理念，提供了自动配置、起步依赖和内嵌服务器三大核心能力，使开发者无需手动管理繁杂的 XML 配置文件即可快速构建生产级别的 Java 应用。

在本系统中，Spring Boot 3.2.5 作为所有业务模块（auth、system、sales、finance）的底层框架，提供了以下关键支撑：一是通过起步依赖（Starter）机制统一管理各组件的版本兼容性，避免了手动引入依赖时的版本冲突问题；二是通过内嵌 Tomcat 容器使每个微服务模块均可独立运行和调试，无需额外部署外部应用服务器；三是通过 `application.yml` 配置文件实现了各模块的数据库连接、服务端口、Nacos 注册中心等参数的集中管理。

#### 4.1.2 微服务治理：Spring Cloud 2023.0.3 + Spring Cloud Alibaba 2023.0.3.2

Spring Cloud 是 Spring 社区提供的一套用于构建分布式系统的工具集，它基于 Spring Boot 提供了服务发现、配置管理、负载均衡、熔断降级等微服务治理能力。Spring Cloud Alibaba 则是阿里巴巴开源的 Spring Cloud 子项目，将阿里系的中间件（如 Nacos、Sentinel）纳入了 Spring Cloud 体系。

在本系统中，Spring Cloud 与 Spring Cloud Alibaba 共同承担了微服务架构的基础治理工作：

- **Nacos（服务注册与配置中心）**：Nacos 是阿里巴巴开源的一个动态服务发现、配置管理和服务管理平台。在本系统中，所有微服务模块（gateway、auth、system、sales、finance）在启动时均向 Nacos 注册自己的服务名与网络地址。当前开发阶段，OpenFeign 客户端采用直接 URL 模式（`url = "http://localhost:XXXX"`），各模块通过硬编码端口直接通信，Nacos 服务发现能力已就绪，可在部署时切换为基于服务名的负载均衡调用模式。同时，Nacos 也作为配置中心统一管理各模块的运行参数。

#### 4.1.3 API 网关：Spring Cloud Gateway

Spring Cloud Gateway 是 Spring 社区基于 Spring WebFlux（响应式编程模型）开发的 API 网关组件，用于替代传统的 Zuul 网关。它基于 Project Reactor 的响应式编程模型，具有非阻塞、高吞吐量的特性。

在本系统中，Gateway 模块（端口 8086）承担了所有外部请求的统一入口角色，具体实现了以下功能：

- **路由转发**：Gateway 中配置了所有后端服务的路由规则，例如 `/api/customer/**` 的请求被转发至 sales 模块（端口 8083），`/api/loanAudit/**` 的请求被转发至 finance 模块（端口 8084）等。前端只需与 Gateway 单一端点通信，无需关心后端微服务的具体地址，从而实现了前后端之间的松耦合。
- **全局鉴权过滤器（AuthFilter）**：Gateway 中实现了一个优先级为 `Order=-100` 的全局过滤器，该过滤器会从请求头中提取 `Authorization: Bearer <token>` 中的 token（即用户 ID），然后调用 auth 模块验证用户身份，并将解析出的用户 ID 与角色信息通过 `X-User-Id` 和 `X-User-Roles` 请求头注入下游服务，使得下游业务模块无需重复实现鉴权逻辑。
- **跨域处理（CORS）**：Gateway 统一配置了跨域资源访问策略，允许来自前端开发服务器（如 `localhost:3000`、`localhost:3001`）的请求跨域访问后端接口。

#### 4.1.4 远程调用：OpenFeign

OpenFeign 是 Spring Cloud 生态中的声明式 REST 客户端，其核心思想是将 HTTP 远程调用抽象为本地 Java 接口的调用方式。开发者只需定义一个带有 `@FeignClient` 注解的接口，并在接口方法上使用 Spring MVC 注解（如 `@GetMapping`、`@PostMapping`）描述远程 API，OpenFeign 会在运行时动态生成该接口的代理实现，自动完成 HTTP 请求的构造、发送与响应反序列化。

在本系统中，OpenFeign 是跨微服务模块同步通信的核心手段，各模块之间的 Feign 调用关系如下：

| 调用方 | 被调用方 | 用途 |
|--------|----------|------|
| sales | auth | 查询用户详情、获取销售代表列表、按部门/战区获取用户 ID 列表 |
| sales | system | 查询部门详情（含战区 ID）、战区详情、获取系统参数值（如公海天数） |
| sales | finance | 创建服务费记录、创建提成记录、查询金融产品详情 |
| finance | auth | 查询用户详情（获取销售代表姓名等信息） |
| finance | sales | 创建业绩记录、更新合同状态、获取合同详情、按状态分页查询合同 |

以"合同详情查询"为例，sales 模块在查询合同时，需要同时展示客户姓名、销售代表姓名、部门名称、战区名称和产品名称，而这些数据分别存储在 auth（用户）、system（部门/战区）和 finance（产品）各自的数据库中。通过 OpenFeign，sales 模块的 `ContractServiceImpl.getDetailWithNames()` 方法在查询到合同基本信息后，依次调用 `authFeignClient.getUserById()` 获取销售代表真实姓名、`systemFeignClient.getDepartmentById()` 获取部门名称、`systemFeignClient.getZoneById()` 获取战区名称、`financeFeignClient.getById()` 获取产品名称，最终将所有数据组装为 `ContractDetailVO` 对象统一返回前端。整个过程对前端透明，前端只需调用一个接口即可获得完整的合同详情数据。

#### 4.1.5 异步消息：RabbitMQ

RabbitMQ 是一个基于 AMQP（高级消息队列协议）的开源消息中间件，由 Rabbit Technologies 公司开发，后由 Pivotal 收购。它支持多种消息模式（如点对点、发布/订阅、路由等），并提供消息持久化、消费者确认、死信队列等高级特性，广泛应用于分布式系统中的异步解耦、削峰填谷和事件驱动架构等场景。

在本系统中，RabbitMQ 用于实现 sales 模块与 finance 模块之间的异步事件驱动通信：

- **合同签署事件（`contract.signed`）**：当 sales 模块中的销售代表签署一份合同后，`ContractSignServiceImpl.sign()` 方法在更新合同状态为"已签署"的同时，构造一个 `ContractSignedEvent` 事件对象（包含合同 ID、客户 ID、销售代表 ID、部门 ID、合同金额、签署日期等关键字段），通过 `RabbitTemplate.convertAndSend()` 方法将事件发送至 RabbitMQ 的 `sales.exchange` 交换机（路由键为 `contract.signed`）。finance 模块中的 `ContractSignedListener` 监听 `contract.signed.queue` 队列，收到事件后自动创建一条贷款审核记录（初始状态为"待接收"），并写入审核轨迹。这种异步方式的优势在于：其一，合同签署操作无需等待金融模块的响应即可返回前端，提升了用户体验；其二，即使金融模块暂时不可用，消息也会暂存于 RabbitMQ 中，待金融模块恢复后重新消费，从而实现了服务间的松耦合与容错。

- **放款批准队列（`loan.approved`）**：系统在 RabbitMQ 中还定义了 `loan.approved.queue` 队列，用于金融审核通过后通知 sales 模块更新合同状态和创建业绩记录。当前实现中，该流程通过 OpenFeign 同步调用完成，但队列已声明就绪，后续可按需切换为异步模式以进一步提升系统韧性。

#### 4.1.6 数据持久化：MyBatis-Plus 3.5.7

MyBatis-Plus 是苞米豆（Baomidou）团队在 MyBatis 基础上开发的一套增强工具，它在不改变 MyBatis 原有功能的前提下，提供了通用 CRUD 操作、分页插件、逻辑删除、自动填充、条件构造器等增强特性，旨在减少重复的 SQL 编写工作，提升开发效率。

在本系统中，MyBatis-Plus 作为所有业务模块的统一 ORM 框架，具体应用了以下增强功能：

- **通用 BaseMapper**：每个数据库表对应的 DAO 接口只需继承 `BaseMapper<T>` 即可自动获得 `insert`、`updateById`、`selectById`、`selectPage`、`deleteById` 等基础 CRUD 方法，无需在 Mapper XML 中编写重复的增删改查 SQL。
- **逻辑删除**：在所有模块中配置了 MyBatis-Plus 的逻辑删除功能（`logic-delete-field: deleted`），当执行 `deleteById()` 方法时，系统不会物理删除数据库中的记录，而是将 `deleted` 字段的值从 0 更新为 1。这种方式保留了数据的可追溯性，也为误删恢复提供了可能。
- **自动填充**：通过 `AutoFillMetaObjectHandler` 实现了 `createdAt`、`updatedAt`、`createdBy`、`updatedBy` 等审计字段的自动填充。这些值在插入或更新记录时由系统自动从当前安全上下文中提取，无需在业务代码中手动设置。
- **分页查询**：使用 MyBatis-Plus 的内置分页插件，结合自定义的 `PageRequest` 和 `PageResponse` 通用 DTO，在所有列表查询接口中实现了统一的分页参数格式和分页响应格式。
- **LambdaQueryWrapper**：在复杂查询场景中，使用 Lambda 表达式构建查询条件，避免了字段名的字符串硬编码，提供了编译期的类型安全检查。

#### 4.1.7 权限体系：Spring Security 6.2.4 + 自定义 RBAC

Spring Security 是 Spring 生态中事实上的安全认证与授权标准框架，由 Spring 社区维护。它提供了一套完整的安全过滤链（Filter Chain），支持基于表单、HTTP Basic、OAuth2、JWT 等多种认证方式，以及基于角色和权限的声明式授权。

在本系统中，auth 模块基于 Spring Security 框架实现了自定义的 RBAC（基于角色的访问控制）权限体系，包含以下五个核心实体：

| 实体 | 说明 |
|------|------|
| sys_user | 系统用户，存储用户名、加密密码、所属部门 ID、所属战区 ID、账户状态等 |
| sys_role | 系统角色，包含角色编码（如 `SUPER_ADMIN`、`DEPT_MANAGER`、`ZONE_DIRECTOR`、`SALES_REP`、`FINANCE_SPECIALIST`）、数据范围级别（1~4）等 |
| sys_user_role | 用户-角色多对多关联表 |
| sys_permission | 系统权限，包含权限编码、权限类型（菜单/按钮/API）、父级 ID（树形结构）、图标、排序等 |
| sys_role_permission | 角色-权限多对多关联表 |

其中，"数据范围"是本系统 RBAC 设计的一个关键特性，共分为四个级别：

- **级别 1（仅本人）**：SALES_REP（销售代表）、FINANCE_SPECIALIST（金融专员）、ACCOUNTANT（会计）只能查看和处理自己名下或自己创建的数据。
- **级别 2（本部门）**：DEPT_MANAGER（部门经理）可以查看和处理本部门下所有成员的数据。
- **级别 3（本战区）**：ZONE_DIRECTOR（战区总监）可以查看和处理本战区下所有部门的数据。
- **级别 4（全部数据）**：SUPER_ADMIN（超级管理员）、GENERAL_MANAGER（总经理）可以查看和处理系统中的全部数据。

数据范围的具体实现采用"服务层过滤"模式。前端在发起列表查询请求时，会从 Pinia Store 中读取当前登录用户的角色信息，将 `filterRole`（角色编码，如 `ROLE_SALES_REP`）、`userId`、`deptId`、`zoneId` 等参数随请求一同发送至后端。后端 Service 层（如 `CustomerServiceImpl.pageList()`、`ContractServiceImpl.pageList()`）根据 `filterRole` 参数，使用 MyBatis-Plus 的 `LambdaQueryWrapper` 动态构建查询条件：若角色为 `ROLE_SALES_REP` 则添加 `eq(Entity::getSalesRepId, userId)` 条件，若为 `ROLE_DEPT_MANAGER` 则添加 `eq(Entity::getDeptId, deptId)` 条件，若为 `ROLE_ZONE_DIRECTOR` 则添加 `eq(Entity::getZoneId, zoneId)` 条件，若为 `ROLE_SUPER_ADMIN` 则不添加任何限制。同时，common 模块中的 `DataScopeAspect` 切面会在 Controller 方法执行前从 SecurityContext 中提取用户信息并存入 `DataScopeContext`（ThreadLocal），为后续可能的 SQL 级别过滤预留了扩展能力。

#### 4.1.8 缓存与会话：Redis

Redis（Remote Dictionary Server）是一个由 Salvatore Sanfilippo 开发的基于内存的键值存储系统，支持字符串、哈希、列表、集合、有序集合等多种数据结构，以其极高的读写性能和丰富的数据结构支持而广泛用于缓存、会话管理和分布式锁等场景。

在本系统中，Redis 部署于 system 模块，用于以下场景：

- **系统参数缓存**：system 模块中的 `sys_param` 表存储了诸如公海客户天数（`customer.public_sea_days`）、提成比率（`performance.commission_rate`）、首期服务费率（`contract.service_fee_first_rate`）等关键业务参数。这些参数变更频率低、读取频率高，使用 Redis 缓存可以减少数据库的重复查询压力。
- **会话与令牌管理**：用户登录后的 token 信息可存储于 Redis 中，通过设置过期时间实现会话的自动失效。

#### 4.1.9 定时任务：Spring @Scheduled

Spring 框架内置的 `@Scheduled` 注解提供了声明式的定时任务调度能力，支持 Cron 表达式、固定频率和固定延迟三种触发模式。在本系统中，`@Scheduled` 用于实现公海客户的自动调入机制：sales 模块的 `PublicSeaTask` 类中定义了一个 Cron 表达式为 `0 0 2 * * ?`（每天凌晨 2 点执行）的定时任务，该任务调用 `CustomerService.listCustomerToPublicSea()` 方法查询所有满足"超过 N 天未跟进（通过 `customer.public_sea_days` 系统参数配置，默认 30 天）且状态为正常"条件的客户，将其状态批量更新为"公海客户"（status=5），从而实现了客户资源的自动流转与再分配。

#### 4.1.10 开发辅助工具

- **Lombok 1.18.32**：由 Reinhard Pötz 等人开发，通过编译期注解（如 `@Data`、`@Builder`、`@AllArgsConstructor`）自动生成实体类的 Getter、Setter、构造器、equals/hashCode 等样板方法，减少手写冗余代码，使实体类代码简洁清晰。
- **Jackson 2.15.4**：由 FasterXML 团队开发的高性能 JSON 处理库，负责 Java 对象与 JSON 之间的序列化与反序列化。本系统通过 `JavaTimeModule` 模块支持了 Java 8 时间类型（如 `LocalDateTime`）的正确序列化。
- **Maven**：由 Apache 软件基金会维护的项目构建与依赖管理工具。本系统采用多模块 Maven 工程（父 POM 定义公共依赖版本，子模块继承使用），统一了 Java 21 编译目标与各依赖的兼容版本。

### 4.2 前端

#### 4.2.1 基础框架：RuoYi-Vue 3.9.2

RuoYi-Vue 是由若依（RuoYi）团队开发的开源后台管理系统脚手架，基于 MIT 协议开源。它整合了 Vue 3、Element Plus、Vue Router、Pinia、Vite 等现代前端技术栈，提供了开箱即用的后台管理模板，包括登录认证、动态路由、权限指令、代码生成、数据字典等通用功能。

在本系统中，前端项目基于 RuoYi-Vue 3.9.2 进行了业务化改造，保留了其基础架构（路由系统、权限体系、请求封装等），并在此基础上开发了销售管理、金融审核、业绩汇总等业务模块的页面组件。

#### 4.2.2 渐进式框架：Vue 3.5.26

Vue.js 是由尤雨溪（Evan You）创建的开源渐进式 JavaScript 框架，采用 Composition API（组合式 API）作为推荐的代码组织方式。与 Vue 2 的 Options API 相比，Composition API 通过 `setup()` 函数或 `<script setup>` 语法糖将相关的逻辑聚合在一起，而非分散在 `data`、`methods`、`computed` 等不同选项中，使得组件的逻辑复用和组织更加灵活。

在本系统中，每个业务页面组件均采用 Vue 3 的 `<script setup>` 语法，通过 `ref` 和 `reactive` 管理组件内部状态，通过 `onMounted` 生命周期钩子触发初始数据加载，通过 `computed` 计算属性实现派生数据的响应式更新。

#### 4.2.3 UI 组件库：Element Plus 2.13.1 + @element-plus/icons-vue 2.3.2

Element Plus 是由 Element 社区基于 Vue 3 重构的企业级 UI 组件库（原 Element UI 仅支持 Vue 2），提供了表格（`el-table`）、表单（`el-form`）、对话框（`el-dialog`）、分页（`el-pagination`）、树形控件（`el-tree`）、选择器（`el-select`）、日期选择器（`el-date-picker`）等 70 余个高质量组件，覆盖了后台管理系统中常见的 UI 交互场景。

在本系统中，Element Plus 是前端页面的核心 UI 构建块：所有列表页面使用 `el-table` 展示数据并结合 `el-pagination` 实现分页浏览；新增和编辑功能使用 `el-dialog` 弹出表单框，表单内部使用 `el-form` 进行字段布局与校验；筛选区域使用 `el-select` 提供条件下拉选择，使用 `el-date-picker` 提供日期范围选择；确认操作使用 `el-message-box` 弹出二次确认对话框。

#### 4.2.4 数据可视化：ECharts 5.6.0

ECharts 是由百度开源、后由 Apache 软件基金会维护的企业级数据可视化图表库。它基于 Canvas 或 SVG 渲染引擎，支持折线图、柱状图、饼图、散点图、漏斗图、仪表盘、热力图等数十种图表类型，并提供数据缩放、图例切换、提示框、 toolbox 等丰富的交互组件。

在本系统中，ECharts 集成于业绩汇总和业绩排名两个模块中。在业绩汇总页面，使用柱状图展示不同销售人员、部门或战区的业绩金额对比；在业绩排名页面，使用横向柱状图或排行榜样式展示按合同金额、提成金额或合同数量排序的销售代表名单。

#### 4.2.5 状态管理：Pinia 3.0.4

Pinia 是由 Vue 核心团队成员 Eduardo San Martin Morote 开发的 Vue 状态管理库，是 Vuex 5 的正式替代方案。它支持模块化的 Store 定义、完整的 TypeScript 类型推导、DevTools 调试支持和热更新等特性。

在本系统中，Pinia 主要用于管理用户认证信息（token、用户 ID、角色列表、权限码列表）和系统全局配置（侧边栏折叠状态、页面尺寸等）。登录成功后，用户信息被存储至 Pinia Store 中，各业务组件通过 Pinia 的 `storeToRefs` 或 `useStore()` 获取当前登录用户的基本信息，据此进行前端层面的权限判断（如按钮的显示与隐藏）。

#### 4.2.6 路由管理：Vue Router 4.6.4 + 动态路由

Vue Router 是 Vue.js 官方的路由管理库，负责管理前端 SPA（单页面应用）中的页面跳转与组件渲染。本系统中的路由采用了"动态路由"模式：前端代码中不硬编码完整的路由表，而是在用户登录成功后，由前端调用 auth 模块的 `/auth/getRouters` 接口，后端根据当前用户被授予的权限码，从 `sys_permission` 表中查询该用户有权访问的菜单项，按树形结构组装为 Vue Router 格式的 JSON 数据返回前端。前端接收到该数据后，通过 `router.addRoute()` 动态注册路由，实现"不同角色看到不同菜单"的效果。

#### 4.2.7 HTTP 通信：Axios 1.15.0

Axios 是由 Matt Zabriskie 开发的基于 Promise 的 HTTP 客户端库，支持在浏览器和 Node.js 环境中运行。它提供了请求/响应拦截器、请求取消、自动 JSON 转换、超时设置和 XSRF 防护等功能。

在本系统中，Axios 被封装为 `src/utils/request.js` 模块，实现了以下通用能力：一是请求拦截器中自动在请求头添加 `Authorization: Bearer <token>`，使得每个 API 请求都携带用户身份凭证；二是响应拦截器中统一处理 HTTP 错误码（如 401 未登录时自动跳转登录页）和业务错误码；三是通过 `NProgress` 进度条库在前端页面顶部展示请求的加载进度。

#### 4.2.8 构建工具：Vite 6.4.2

Vite 是由尤雨溪团队开发的下一代前端构建工具，它利用浏览器原生 ES Module 支持实现了开发服务器的极速冷启动（无需打包整个应用），并通过 Rollup 打包实现了生产环境构建的高度优化。与传统基于 Webpack 的工具链相比，Vite 的开发服务器启动速度快一个数量级，HMR（模块热替换）几乎即时生效。

本系统使用 Vite 6.4.2 作为前端项目的构建工具，并集成了以下 Vite 插件：

- **@vitejs/plugin-vue**：提供 Vue 3 SFC（单文件组件）的编译支持。
- **vite-plugin-compression**：在构建时自动生成 gzip 压缩后的静态资源文件，配合 Nginx 的 gzip_static 模块可减少网络传输体积。
- **vite-plugin-svg-icons**：将 SVG 图标文件转换为雪碧图，减少图标加载的 HTTP 请求数。
- **unplugin-auto-import**：自动导入 Vue 3 Composition API（如 `ref`、`computed`、`onMounted` 等），使开发者无需在每个 `.vue` 文件中手动书写 `import { ref } from 'vue'`。

### 5.1 后端实现

#### 5.1.1 系统管理

##### 用户管理

**前置依赖**：auth 模块需接入 MySQL 数据库（`dafuweng_auth`）中的 `sys_user` 表、`sys_user_role` 关联表、`sys_role` 表；密码加密依赖 Spring Security 的 `BCryptPasswordEncoder`。

**实现步骤**：

- **用户列表查询**：
  1. 前端发送 GET 请求至 `/api/sysUser/page`，携带分页参数（`page`、`size`）和可选的筛选条件（用户名、部门 ID 等）。
  2. Gateway 将请求转发至 auth 模块的 `SysUserController.pageList()` 方法。
  3. Controller 调用 `SysUserService.pageList()`，该方法构造 `LambdaQueryWrapper` 查询条件（若有关键词搜索则添加 `LIKE` 条件），排除已逻辑删除的记录（`deleted=0`），通过 MyBatis-Plus 的分页插件执行分页查询。
  4. 后端将查询结果封装为 `PageResponse<SysUserEntity>` 统一返回格式，通过 Result 包装后返回前端。
  5. 前端 `el-table` 组件渲染用户列表，`el-pagination` 组件渲染分页控件。

- **新增用户**：
  1. 前端弹出新增对话框，用户填写用户名、真实姓名、密码、手机号、邮箱、选择所属部门与战区后点击确定。
  2. 前端发送 POST 请求至 `/api/sysUser`，请求体为 `SysUserEntity` 的 JSON 表示。
  3. auth 模块的 `SysUserController.save()` 接收请求，调用 `SysUserService.save()`。
  4. `SysUserService.save()` 首先校验用户名是否已存在（通过 `LambdaQueryWrapper` 查询 `sys_user` 表中 `username` 和 `deleted=0` 的记录），若存在则抛出 `IllegalArgumentException`，前端收到 500 错误码并显示"用户名已存在"提示；若不存在则使用 `BCryptPasswordEncoder` 对明文密码进行加密，设置 `deleted=0`，调用 `sysUserDao.insert()` 将记录写入 `sys_user` 表。
  5. 返回成功，前端关闭对话框并刷新列表。

- **分配角色**：
  1. 前端在用户列表的操作列提供"分配角色"按钮，点击后弹出角色分配对话框，展示所有可用角色（通过调用角色列表接口获取）的多选框。
  2. 用户勾选目标角色后点击确定，前端发送 PUT 请求至 `/api/sysUser/{id}/roles`，请求体为 `{ "roleIds": [1, 3, 5] }`。
  3. auth 模块的 `SysUserController.assignRoles()` 接收请求，调用 `SysUserService.assignRoles()`。
  4. `assignRoles()` 方法先删除 `sys_user_role` 表中该用户的所有现有角色关联记录，然后遍历传入的 `roleIds` 列表，逐条插入新的用户-角色关联记录。该操作使用 `@Transactional` 注解保证原子性。
  5. 返回成功，前端提示"角色分配成功"。

- **禁用/启用用户**：
  1. 前端发送 PUT 请求至 `/api/sysUser`，请求体中包含 `id` 和 `status` 字段（`status=0` 为禁用，`status=1` 为启用）。
  2. auth 模块的 `SysUserController.update()` 调用 `SysUserService.update()`，通过 MyBatis-Plus 的 `updateById()` 更新用户状态字段。
  3. 禁用的用户将无法登录系统（登录时校验 `status` 字段）。

- **解锁用户**：
  1. 当用户连续输入错误密码超过系统配置的最大次数（由 `sys_param` 中的 `login.max_error_times` 参数控制，默认为 5 次）时，账户将被锁定（`lock_time` 字段设置锁定时间戳）。
  2. 管理员可在前端用户列表点击"解锁"按钮，前端发送 PUT 请求至 `/api/sysUser/{id}/unlock`。
  3. auth 模块将用户的 `login_error_count` 重置为 0，`lock_time` 置为 `null`，恢复用户登录权限。

**核心代码——用户登录（`SysUserServiceImpl.login()`）**：

```java
public SysUserEntity login(String username, String password, String loginIp) {
    SysUserEntity user = sysUserDao.selectByUsername(username);
    if (user == null) {
        throw new IllegalArgumentException("用户名或密码错误");
    }
    // 锁定检测
    if (user.getLockTime() != null && user.getLockTime().after(new Date())) {
        throw new IllegalArgumentException("账号已锁定，请稍后再试");
    }
    // BCrypt 密码校验
    if (!passwordEncoder.matches(password, user.getPassword())) {
        int errors = (user.getLoginErrorCount() == null ? 0 : user.getLoginErrorCount()) + 1;
        user.setLoginErrorCount(errors);
        if (errors >= 5) {
            Calendar cal = Calendar.getInstance();
            cal.add(Calendar.MINUTE, 30);          // 5 次错误锁定 30 分钟
            user.setLockTime(cal.getTime());
        }
        sysUserDao.updateById(user);
        throw new IllegalArgumentException("用户名或密码错误");
    }
    // 登录成功：重置错误计数，记录登录时间和 IP
    sysUserDao.update(null,
        new LambdaUpdateWrapper<SysUserEntity>()
            .eq(SysUserEntity::getId, user.getId())
            .set(SysUserEntity::getLoginErrorCount, 0)
            .set(SysUserEntity::getLastLoginTime, new Date())
            .set(SysUserEntity::getLastLoginIp, loginIp));
    user.setPassword(null);  // 不返回密码哈希
    return user;
}
```

**核心代码——新增用户（`SysUserServiceImpl.save()`）**：

```java
public SysUserEntity save(SysUserEntity entity) {
    // 用户名唯一性校验
    if (StringUtils.hasText(entity.getUsername())
            && sysUserDao.countByUsername(entity.getUsername()) > 0) {
        throw new IllegalArgumentException("用户名已存在");
    }
    // BCrypt 加密密码
    if (StringUtils.hasText(entity.getPassword())) {
        entity.setPassword(passwordEncoder.encode(entity.getPassword()));
    }
    entity.setDeleted((short) 0);
    sysUserDao.insert(entity);
    return entity;
}
```

##### 角色管理

**前置依赖**：auth 模块的 `sys_role` 表、`sys_role_permission` 关联表、`sys_permission` 表。

**实现步骤**：

- **角色列表查询**：与用户列表类似，前端调用 `/api/sysRole/page` 接口，后端基于 MyBatis-Plus 分页查询 `sys_role` 表并返回。
- **新增角色**：前端发送 POST 请求至 `/api/sysRole`，包含角色编码、角色名称、数据范围（1~4）、排序号、状态等字段。后端校验角色编码唯一性后写入 `sys_role` 表。
- **分配角色权限**：前端在角色列表的操作列提供"分配权限"按钮，点击后弹出权限树（通过调用权限列表接口获取全部权限的树形结构），管理员勾选该角色拥有的权限后点击确定。前端发送 PUT 请求至 `/api/sysRole/{id}/permissions`，请求体为权限 ID 列表。后端先删除 `sys_role_permission` 表中该角色的所有现有权限关联记录，再批量插入新的关联记录。此处的"权限"可以是菜单级别的（决定用户能看到哪些菜单）或按钮/API 级别的（决定用户在页面中能执行哪些操作）。
- **禁用角色**：将角色的 `status` 字段更新为 0，被禁用的角色将不再出现在用户角色分配的候选列表中。

##### 权限管理

**前置依赖**：auth 模块的 `sys_permission` 表（树形结构，通过 `parent_id` 字段建立父子关系）。

**实现步骤**：

- **权限列表查询**：前端调用 `/api/sysPermission/page` 接口，后端返回权限的分页列表。同时，前端可调用权限树接口获取完整的树形结构用于角色权限分配。
- **新增权限**：前端发送 POST 请求至 `/api/sysPermission`，包含权限编码（如 `SALES_CUSTOMER_LIST`）、权限名称（如"客户列表"）、权限类型（菜单/按钮/API）、路由路径、父级 ID、图标、排序号等字段。后端将新权限写入 `sys_permission` 表。若设置了父级 ID，该权限将作为子节点挂在对应父权限的树形结构下。
- **禁用权限**：将权限的 `status` 字段更新为 0。被禁用的权限在 `getRouters` 接口中将不再被包含在返回的路由树中，从而实现了菜单项的前后端双重控制。

**核心代码——权限删除（递归删除子节点）**：

```java
@Transactional
public void delete(Long id) {

    sysPermissionDao.deleteById(id);

private void deleteRecursively(Long parentId) {
    List<SysPermissionEntity> children = listByParentId(parentId);
    if (children != null && !children.isEmpty()) {
        for (SysPermissionEntity child : children) {
            deleteRecursively(child.getId());
            sysPermissionDao.deleteById(child.getId());
        }
    }
}
```

**前置依赖**：system 模块的 `sys_department` 表和 `sys_zone` 表；Redis（用于缓存查询结果）。

**实现步骤**：

- **部门列表查询**：前端调用 `/api/sysDepartment/page` 接口。system 模块的 `SysDepartmentServiceImpl.pageList()` 方法通过 MyBatis-Plus 分页查询 `sys_department` 表，返回部门列表。在构建响应数据时，对于每个部门的 `zone_id` 字段，service 层通过 `SysZoneDao.selectById()` 查询战区名称，将战区名称填充到返回数据中，避免了前端仅看到一个战区 ID 而无法理解其含义的问题。

**前置依赖**：system 模块的 `sys_param` 表；Redis 缓存。

- **跨模块使用**：系统参数不仅被 system 模块本身使用，也被其他模块通过 OpenFeign 远程读取。例如，sales 模块的 `PublicSeaTask` 定时任务通过 `SystemFeignClient.getParamValue("customer.public_sea_days")` 获取公海客户的天数阈值；finance 模块的提成计算通过系统参数获取服务费率。

#### 5.1.2 销售服务

销售服务模块由 sales 微服务承担，是整个业务系统的核心。

##### 客户管理

**前置依赖**：sales 模块的 `customer` 表；auth 模块的用户查询接口（通过 OpenFeign 调用，用于获取销售代表列表和查询用户详情）；system 模块的部门/战区查询接口。

**实现步骤**：

- **客户列表查询（含数据范围过滤）**：
  1. 前端发送 GET 请求至 `/api/customer/page`，携带分页参数和可选的 `filterRole`、`userId`、`deptId`、`zoneId` 参数（这些参数由前端从 Pinia Store 中读取当前登录用户信息后自动附加）。
  2. sales 模块的 `CustomerController.pageList()` 接收请求，调用 `CustomerService.pageList()`。
  3. `pageList()` 方法根据 `filterRole` 参数动态构建数据过滤条件：若为 `SUPER_ADMIN` 则不添加任何过滤（可查看全部客户）；若为 `ZONE_DIRECTOR` 则添加 `zone_id = {当前用户战区ID}` 条件；若为 `DEPT_MANAGER` 则添加 `dept_id = {当前用户部门ID}` 条件；若为 `SALES_REP` 则添加 `sales_rep_id = {当前用户ID}` 条件。同时，所有查询均排除状态为"公海客户"（status=5）的记录和已逻辑删除的记录。
  4. 通过 MyBatis-Plus 分页插件执行查询，将结果封装为 `PageResponse` 返回前端。

**核心代码——客户分页查询（含数据范围过滤）**：


##### 公海客户

**前置依赖**：sales 模块的 `customer` 表（含 `status`、`last_contact_date`、`public_sea_time` 等公海相关字段）和 `customer_transfer_log` 表；system 模块的系统参数接口（获取公海天数阈值）；auth 模块的用户查询接口。

**实现步骤**：

- **客户自动调入公海（定时任务）**：
  1. sales 模块的 `PublicSeaTask` 类中定义了一个 `@Scheduled(cron = "0 0 2 * * ?")` 定时任务，每天凌晨 2 点自动触发执行。
  2. 任务方法 `scanPublicSeaCustomers()` 首先通过 `SystemFeignClient.getParamValue("customer.public_sea_days")` 从 system 模块读取公海天数阈值（若读取失败则使用默认值 30 天）。
  3. 调用 `CustomerService.listCustomerToPublicSea(publicSeaDays)` 方法。该方法执行自定义 SQL 查询：筛选 `status` 不为"公海客户"（status=3,4,5）的非特殊状态客户，且其 `last_contact_date` 距离当前时间已超过阈值天数，且客户创建时间早于"当前时间 - 阈值天数"（防止新建客户立即被调入公海）。
  4. 对于查询到的每一名符合条件的客户，将其 `status` 字段更新为 5（公海客户），`public_sea_time` 字段更新为当前时间，调用 `customerDao.updateById()` 持久化。
  5. 进入公海的客户不再归属原来的销售代表，出现在公海客户列表中供其他符合条件的销售人员认领或分配。

**核心代码——公海定时扫描任务（`PublicSeaTask.scanPublicSeaCustomers()`）**：

```java
@Scheduled(cron = "0 0 2 * * ?")   // 每天凌晨 2 点执行
public void scanPublicSeaCustomers() {
    Integer publicSeaDays = getPublicSeaDays();  // 从 system 模块读取阈值，默认 30
    List<CustomerEntity> toPublicSea = customerService.listCustomerToPublicSea(publicSeaDays);
    for (CustomerEntity customer : toPublicSea) {
        customer.setStatus((short) 5);            // 设置为公海客户
        customer.setPublicSeaTime(new Date());
        customerService.update(customer);
    }
}

private Integer getPublicSeaDays() {
    try {
        Result<String> result = systemFeignClient.getParamValue("customer.public_sea_days");
        if (result != null && result.getData() != null) {
            return Integer.parseInt(result.getData());
        }
        return 30;  // 读取失败时使用默认值
    } catch (Exception e) { return 30; }
}
```

- **跟进后自动重置公海计时**：
  1. 当销售代表对名下客户添加跟进记录时（见 5.2.3 节），系统设计上会在保存跟进记录后同步更新对应 `customer` 表中该客户的 `last_contact_date` 字段为当天日期（该字段已在 `CustomerEntity` 中定义，更新逻辑通过 Controller 层在调用 `ContactRecordService.save()` 之后同步执行 `CustomerService.update()` 完成）。
  2. 由于定时任务的判断条件依赖于 `last_contact_date` 与当前时间的差值，更新该字段即相当于"重置了计时器"，该客户在下一次定时任务扫描时将不会被判定为符合公海调入条件。
  3. 对于已经是公海客户（status=5）的客户，在通过公海分配重新分配给销售代表后，其 `status` 变更为正常客户状态，且 `last_contact_date` 被重置，重新开始计时。

- **公海客户分配**：
  1. 战区总监、销售经理或销售代表在前端公海客户页面查看到公海客户列表（仅显示 status=5 的客户）。
  2. 点击某个公海客户的"分配"按钮，前端弹出分配对话框，展示当前操作人管辖范围内的销售代表下拉列表（通过 `/api/customer/public-sea/sales-reps` 接口获取，该接口通过 `AuthFeignClient.listSalesReps()` 获取所有销售代表，然后按当前用户的数据范围——战区 ID 或部门 ID——进行过滤）。
  3. 选择目标销售代表后点击确定，前端发送 PUT 请求至 `/api/customer/public-sea/transfer`，请求体包含 `customerId`、`toRepId`、`reason`（分配原因）、`operatorId`（操作人 ID）、`operateType`（操作类型，如 `ASSIGN`）。
  4. `PublicSeaController.transfer()` 调用 `PublicSeaService.transfer()`，执行以下操作：
     a. 校验客户是否存在且当前状态为公海客户（status=5），若条件不满足则抛出异常。
     b. 权限校验：若操作人为销售代表角色，则只能将公海客户分配给自己（`toRepId` 必须等于 `operatorId`）；若为战区总监或部门经理，则无此限制。
     c. 通过 `AuthFeignClient.getUserById(toRepId)` 获取目标销售代表的所属部门 ID 和战区 ID。
     d. 更新客户记录：将 `salesRepId`、`deptId`、`zoneId` 更新为目标销售代表的信息，将 `status` 从 5（公海客户）更新为 1（正常客户）。
     e. 在 `customer_transfer_log` 表中插入一条转移记录，`operateType` 为 `ASSIGN`（公海分配），`fromRepId` 为 `null`（表示从公海转入）。
  5. 返回成功，分配给销售代表的客户将出现在其个人客户列表中。

**核心代码——公海客户分配（`PublicSeaServiceImpl.transfer()`）**：

```java
@Transactional
public void transfer(Long customerId, Long toRepId, String reason,
                     Long operatorId, String operateType) {
    // 1. 校验客户必须为公海状态（status=5）
    CustomerEntity customer = customerDao.selectById(customerId);
    if (customer == null || customer.getStatus() != 5) {
        throw new RuntimeException("该客户不是公海客户");
    }
    // 2. 权限校验：销售代表只能分配给自己
    if (isSalesRep(operatorId) && !toRepId.equals(operatorId)) {
        throw new RuntimeException("销售代表只能将公海客户转移给自己");
    }
    // 3. 获取目标销售代表的部门 ID 和战区 ID（通过 Feign 远程调用 auth 模块）
    Result<?> repRes = authFeignClient.getUserById(toRepId);
    // ... 从 repRes 中提取 deptId、zoneId ...
    // 4. 更新客户：更换销售代表、部门、战区，状态从 5→1
    customer.setSalesRepId(toRepId);
    customer.setDeptId(deptId);
    customer.setZoneId(zoneId);
    customer.setStatus((short) 1);
    customerDao.updateById(customer);
    // 5. 写入转移日志（fromRepId=null 表示从公海转入）
    CustomerTransferLogEntity log = new CustomerTransferLogEntity();
    log.setCustomerId(customerId);
    log.setFromRepId(null);        // 从公海转入
    log.setToRepId(toRepId);
    log.setOperateType(operateType);
    log.setReason(reason);
    customerTransferLogDao.insert(log);
}
```

```java
public String generateContractNo() {
    String dateStr = LocalDate.now().format(DateTimeFormatter.ofPattern("yyyyMMdd"));
    String prefix = "HT-" + dateStr + "-";
    // 查询当天最大的流水号
    ContractEntity lastContract = contractDao.selectOne(
        new LambdaQueryWrapper<ContractEntity>()
            .likeRight(ContractEntity::getContractNo, prefix)
            .orderByDesc(ContractEntity::getId)
            .last("LIMIT 1"));
    int sequence = 1;
    if (lastContract != null) {
        String lastSeqStr = lastContract.getContractNo()
            .substring(lastContract.getContractNo().lastIndexOf("-") + 1);
        sequence = Integer.parseInt(lastSeqStr) + 1;
    }
    return prefix + String.format("%04d", sequence);  // 如 HT-20260511-0001
}
```

- **合同签署（触发异步事件）**：
  1. 前端在合同详情页面点击"签署合同"按钮，弹出确认对话框。
  2. 前端发送 POST 请求至 `/api/contract/{id}/sign`。
  3. sales 模块的 `ContractSignServiceImpl.sign()` 方法执行以下操作：
     a. 校验合同当前状态必须为"待签署"（status=1），否则抛出状态异常。
     b. 将合同状态更新为"已签署"（status=2），设置签署日期为当前日期，调用 `contractDao.updateById()` 持久化。
     c. 构造 `ContractSignedEvent` 事件对象，包含合同 ID、客户 ID、销售代表 ID、部门 ID、合同金额和签署日期。
     d. 通过 `RabbitTemplate.convertAndSend()` 将事件发送至 RabbitMQ 的 `sales.exchange` 交换机（路由键为 `contract.signed`）。
     e. finance 模块的 `ContractSignedListener` 接收到该事件后，在 `loan_audit` 表中自动创建一条审核记录（audit_status=1，待接收），并写入审核轨迹（operator_id=0，表示系统自动操作）。（注：此步骤在 finance 模块内部完成，对 sales 模块透明。）
  4. 返回成功，前端更新合同状态显示。

**核心代码——合同签署（`ContractSignServiceImpl.sign()`）**：

```java
@Transactional
public void sign(Long contractId) {
    ContractEntity contract = contractDao.selectById(contractId);
    if (contract.getStatus() != 1) {
        throw new IllegalStateException("当前状态不允许签署");
    }
    // 更新合同状态为已签署
    contract.setStatus((short) 2);
    contract.setSignDate(new Date());
    contractDao.updateById(contract);

    // 构造事件并通过 RabbitMQ 发送至 finance 模块
    ContractSignedEvent event = new ContractSignedEvent();
    event.setContractId(contract.getId());
    event.setCustomerId(contract.getCustomerId());
    event.setSalesRepId(contract.getSalesRepId());
    event.setDeptId(contract.getDeptId());
    event.setContractAmount(contract.getContractAmount());
    event.setSignDate(contract.getSignDate());

    rabbitTemplate.convertAndSend(
        MqConfig.EXCHANGE_SALES,           // sales.exchange
        MqConfig.ROUTING_CONTRACT_SIGNED,  // contract.signed
        event
    );
}
```

  3. `ContractServiceImpl.payFirstInstallment()` 方法执行以下操作：
     a. 校验合同当前状态必须为"已签署"（status=2）。
     b. 将合同状态更新为"已付首期"（status=3），标记 `serviceFee1Paid=1`。
     c. 通过 `FinanceFeignClient.createServiceFeeRecord()` 向 finance 模块发送请求，创建首期服务费记录（feeType=1，金额为 `serviceFee1`，支付状态为"已支付"）。
  4. 返回成功。

**核心代码——支付首期（`ContractServiceImpl.payFirstInstallment()`）**：

```java
        throw new IllegalStateException("当前状态不允许操作");
```

  1. 前端点击"提交金融部"按钮，前端发送 POST 请求至 `/api/contract/{id}/submit-to-finance`。
  2. `ContractServiceImpl.submitToFinance()` 方法校验合同状态必须为"已付首期"（status=3），然后将状态更新为"已提交金融"（status=4），设置提交时间为当前时间。

- **银行已放款（自动创建二期服务费记录和提成记录）**：
  1. 当合同在金融审核流程中通过银行放款后，前端调用"银行已放款"接口（或 finance 模块通过 OpenFeign 调用 sales 模块的 `updateContractStatus` 更新状态）。
  2. 前端发送 POST 请求至 `/api/contract/{id}/bank-loan`。
  3. `ContractServiceImpl.bankLoan()` 方法执行以下操作：
     a. 校验合同状态必须为"审核通过"（status=5）。
     b. 计算实际放款金额 = 合同金额 - 首期服务费 - 二期服务费。
     c. 更新合同状态为"已放款"（status=7），标记 `serviceFee2Paid=1`，记录实际放款金额。
     d. 通过 `FinanceFeignClient.createServiceFeeRecord()` 创建二期服务费记录（feeType=2）。
     e. 在 sales 模块本地创建业绩记录（`PerformanceRecordEntity`），提成比例按 1.5% 计算，提成金额 = 合同金额 × 1.5%。
     f. 通过 `FinanceFeignClient.createCommissionRecord()` 在 finance 模块创建对应的提成发放记录。

**核心代码——银行放款（`ContractServiceImpl.bankLoan()`）**：

```java
@Transactional
public void bankLoan(Long id) {
    ContractEntity contract = contractDao.selectById(id);
    if (contract.getStatus() != 5) {
        throw new IllegalStateException("当前状态不允许操作");
    }
    // 计算实际放款金额 = 合同金额 - 首期服务费 - 二期服务费
    BigDecimal actualLoanAmount = contract.getContractAmount()
        .subtract(contract.getServiceFee1() != null ? contract.getServiceFee1() : BigDecimal.ZERO)
        .subtract(contract.getServiceFee2() != null ? contract.getServiceFee2() : BigDecimal.ZERO);
    // 更新合同状态为已放款
    contract.setStatus((short) 7);
    contract.setServiceFee2Paid((short) 1);
    contract.setActualLoanAmount(actualLoanAmount);
    contractDao.updateById(contract);

    // 通过 Feign 在 finance 模块创建二期服务费记录（feeType=2）
    financeFeignClient.createServiceFeeRecord(serviceFeeRecord);

    // 在 sales 本地创建业绩记录（提成比例固定 1.5%）
    BigDecimal commissionAmount = contract.getContractAmount()
        .multiply(new BigDecimal("0.015"));
    PerformanceRecordEntity perf = new PerformanceRecordEntity();
    perf.setContractId(contract.getId());
    perf.setSalesRepId(contract.getSalesRepId());
    perf.setCommissionRate(new BigDecimal("0.015"));
    perf.setCommissionAmount(commissionAmount);
    performanceRecordService.save(perf);

    // 通过 Feign 在 finance 模块创建提成发放记录
    financeFeignClient.createCommissionRecord(commissionRecord);
}
```

#### 5.1.3 金融审核

金融审核模块由 finance 微服务承担。

##### 银行管理

**前置依赖**：finance 模块的 `bank` 表。

**实现步骤**：银行管理的增删改查为标准 CRUD 操作。前端调用 `/api/bank/page` 分页查询银行列表，通过 POST/PUT/DELETE 请求至 `/api/bank` 和 `/api/bank/{id}` 实现新增、修改和删除。银行信息（银行编码、银行名称、支行名称、联系人）在贷款审核的"提交银行"环节被引用，金融专员可从已维护的银行列表中选择目标银行提交贷款申请。

##### 金融产品管理

**前置依赖**：finance 模块的 `finance_product` 表，与 `bank` 表存在外键关联（`bank_id`）。

**实现步骤**：金融产品的增删改查为标准 CRUD 操作。每条金融产品记录包含产品编码、产品名称、所属银行 ID、最小/最大贷款金额、利率、最小/最大贷款期限和提成比率等字段。在合同管理中，销售代表创建合同时需要选择一个金融产品，此时前端通过调用金融产品列表接口获取可选产品。在贷款审核的终审环节，系统根据合同关联的金融产品 ID 获取其提成比率，用于计算销售代表的提成金额。

##### 贷款审核

**前置依赖**：finance 模块的 `loan_audit` 表和 `loan_audit_record`（审核轨迹）表；sales 模块的合同查询接口和业绩创建接口（通过 OpenFeign 调用）；auth 模块的用户查询接口；RabbitMQ 的 `contract.signed.queue`。

**实现步骤**：

- **审核记录自动创建**：
  1. 当 sales 模块签署合同后发送 `ContractSignedEvent` 至 RabbitMQ。
  2. finance 模块的 `ContractSignedListener.onContractSigned()` 方法接收事件。
  3. 该方法首先检查该合同是否已存在审核记录（通过 `loanAuditService.getByContractId(contractId)` 查询），若已存在则直接返回（保证幂等性，防止重复消费导致重复创建）。
  4. 若不存在，则创建一条新的 `LoanAuditEntity`，设置 `contractId` 为事件中的合同 ID，`auditStatus` 为 1（待接收），写入 `loan_audit` 表。
  5. 同时写入一条审核轨迹记录（`LoanAuditRecordEntity`），`action` 为 `receive`，`operatorId` 为 0（系统自动），`content` 为"合同签署自动创建审核任务"。

**核心代码——合同签收监听器（`ContractSignedListener.onContractSigned()`）**：

```java
@RabbitListener(queues = MqConfig.QUEUE_CONTRACT_SIGNED)
public void onContractSigned(ContractSignedEvent event) {
    // 幂等性检查：若已存在该合同的审核记录则直接返回
    LoanAuditEntity existing = loanAuditService.getByContractId(event.getContractId());
    if (existing != null) return;

    // 创建贷款审核记录（audit_status=1，待接收）
    LoanAuditEntity audit = new LoanAuditEntity();
    audit.setContractId(event.getContractId());
    audit.setAuditStatus((short) 1);
    loanAuditService.save(audit);

    // 写入审核轨迹（operator_id=0 表示系统自动操作）
    LoanAuditRecordEntity record = new LoanAuditRecordEntity();
    record.setLoanAuditId(audit.getId());
    record.setAction("receive");
    record.setOperatorId(0L);
    record.setOperatorName("系统");
    record.setOperatorRole("system");
    record.setContent("合同签署自动创建审核任务");
    loanAuditRecordService.save(record);
}
```

- **贷款审核列表（仅显示审核中的合同）**：
  1. 前端发送 GET 请求至 `/api/loanAudit/page`。
  2. finance 模块的 `LoanAuditServiceImpl.pageList()` 方法查询 `loan_audit` 表中所有未删除的审核记录，并通过自定义 Mapper XML 的 `selectPageWithNames` 方法进行关联查询。
  3. `fillNames()` 私有方法通过 `SalesFeignClient.getContractById()` 获取合同信息，从中提取客户 ID；通过 `AuthFeignClient.getUserById()` 获取销售代表真实姓名，将客户标识和销售代表姓名填充进审核记录中，使前端列表能够展示可读的关联信息。

- **审核流程（状态机流转）**：
  1. **接收合同**：金融专员在前端点击"接收"，前端发送 POST 请求至 `/api/loanAudit/{id}/receive`。后端校验当前审核状态必须为 1（待接收），状态更新为 2（待初审），写入审核轨迹。
  2. **初审**：金融专员完成材料初审后点击"初审通过"，前端发送 POST 请求至 `/api/loanAudit/{id}/review`。后端校验当前状态为 2，更新为 3（待提交银行），记录审核日期。
  3. **提交银行**：金融专员选择目标银行后点击"提交银行"，前端发送 POST 请求至 `/api/loanAudit/{id}/submit-bank`，请求体包含 `bankId`。后端校验当前状态为 3，更新为 4（待银行反馈），记录银行申请时间。
  4. **银行反馈**：收到银行反馈后，前端发送 POST 请求至 `/api/loanAudit/{id}/bank-result`，请求体包含 `approved`（布尔值）和银行反馈内容。若银行通过，状态更新为 6（银行通过）；若银行拒绝，状态更新为 5（银行拒绝）。
  5. **终审批准**（仅银行通过后可执行）：金融专员在终审页面填写实际放款金额、实际利率和放款日期后点击"批准"，前端发送 POST 请求至 `/api/loanAudit/{id}/approve`。后端执行以下操作：
     a. 校验当前审核状态必须为 6（银行通过）。
     b. 更新审核记录中的实际放款金额、利率和放款日期。
     c. 通过 `SalesFeignClient.getContract(audit.getContractId())` 获取合同信息。
     d. 根据合同关联的金融产品 ID，通过 `FinanceProductService.getById()` 获取产品的提成比率，计算提成金额 = 合同金额 × 提成比率。
     e. 构造 `PerformanceCreateDTO` 对象，通过 `SalesFeignClient.createPerformance(perfDto)` 向 sales 模块创建业绩记录。
     f. 通过 `SalesFeignClient.updateContractStatus()` 将合同状态更新为 7（已放款）。

**核心代码——终审批准（`LoanAuditServiceImpl.approve()`）**：

```java
@Transactional
public void approve(Long id, Long operatorId, ..., BigDecimal actualLoanAmount,
                    BigDecimal actualInterestRate, Date loanGrantedDate) {
    LoanAuditEntity audit = loanAuditDao.selectById(id);
    if (audit.getAuditStatus() != 6) {
        throw new IllegalStateException("当前状态不允许终审");
    }
    audit.setActualLoanAmount(actualLoanAmount);
    audit.setActualInterestRate(actualInterestRate);
    audit.setLoanGrantedDate(loanGrantedDate);
    loanAuditDao.updateById(audit);
    saveRecord(id, "approve", operatorId, operatorName, operatorRole, comment);

    // 通过 Feign 获取合同信息
    ContractVO contract = salesFeignClient.getContract(audit.getContractId()).getData();

    // 根据产品提成比率计算提成金额
    PerformanceCreateDTO perfDto = new PerformanceCreateDTO();
    perfDto.setContractId(audit.getContractId());
    perfDto.setSalesRepId(contract.getSalesRepId());
    perfDto.setContractAmount(contract.getContractAmount());
    if (contract.getProductId() != null) {
        FinanceProductEntity product = financeProductService.getById(contract.getProductId());
        if (product != null && product.getCommissionRate() != null) {
            perfDto.setCommissionRate(product.getCommissionRate());
            perfDto.setCommissionAmount(
                contract.getContractAmount().multiply(product.getCommissionRate()));
        }
    }
    // 通过 Feign 在 sales 模块创建业绩记录
    salesFeignClient.createPerformance(perfDto);
    // 通过 Feign 更新合同状态为已放款
    salesFeignClient.updateContractStatus(audit.getContractId(), (short) 7);
}
```
  6. **审核拒绝**（loan_audit 工作流内部）**：前端点击"拒绝"，发送 POST 请求至 `/api/loanAudit/{id}/reject`。后端校验当前审核状态为 4（待银行反馈）、5（银行拒绝）或 7 时允许执行，将审核状态更新为 7（终审拒绝），写入审核轨迹。此操作仅更新贷款审核记录的状态，不直接影响合同状态。合同级别的审核拒绝通过独立的 `/api/contractAudit/{contractId}/reject` 接口实现，该接口调用 `SalesFeignClient.updateContractStatusWithReason()` 将合同状态由 4（已提交金融）变更为 6（审核拒绝），并携带拒绝原因。

```
待接收(1) → 待初审(2) → 待提交银行(3) → 待银行反馈(4) → 银行通过(6) → 终审通过(更新合同为已放款)
                                                    → 银行拒绝(5)
                                                                  终审拒绝(7) → 更新合同为已拒绝
```

#### 5.1.4 业绩汇总

业绩汇总模块由 sales 微服务承担（业绩数据存储在 sales 库的 `performance_record` 表中）。

##### 业绩汇总

**前置依赖**：sales 模块的 `performance_record` 表和 `contract` 表；auth 模块的用户查询接口。

**实现步骤**：

- **多维度汇总查询**：
  1. 前端业绩汇总页面提供多维度的筛选条件：销售人员（下拉选择）、部门（下拉选择）、战区（下拉选择）、月份（日期选择器）。
  2. 用户选择筛选条件后点击查询，前端发送 GET 请求至 `/api/perfSummary/summary`，携带所有筛选参数。
  3. sales 模块的 `PerfSummaryServiceImpl` 接收请求后，根据筛选条件构建动态 SQL 查询（使用 MyBatis-Plus 的 `LambdaQueryWrapper` 或自定义 Mapper XML）。
  4. 查询结果按维度分组聚合（如按销售人员分组则 GROUP BY sales_rep_id），汇总每个分组的合同总金额、提成总金额、合同数量等指标。
  5. 返回的汇总数据中，销售代表 ID、部门 ID、战区 ID 通过 OpenFeign 调用 auth 和 system 模块转换为可读的名称。
  6. 前端使用 ECharts 柱状图或表格展示汇总结果。

**核心代码——业绩汇总（`PerfSummaryServiceImpl.summary()`）**：

```java
public Map<String, Object> summary(String beginTime, String endTime,
        Long deptId, Long zoneId, String groupBy) {
    LambdaQueryWrapper<PerformanceRecordEntity> wrapper = new LambdaQueryWrapper<>();
    wrapper.eq(PerformanceRecordEntity::getStatus, 2); // 只查已确认的业绩
    if (deptId != null) wrapper.eq(PerformanceRecordEntity::getDeptId, deptId);
    if (zoneId != null) wrapper.eq(PerformanceRecordEntity::getZoneId, zoneId);
    // 时间范围筛选
    if (StringUtils.hasText(beginTime)) wrapper.ge(..., beginTime + " 00:00:00");
    if (StringUtils.hasText(endTime))   wrapper.le(..., endTime + " 23:59:59");

    List<PerformanceRecordEntity> records = performanceRecordDao.selectList(wrapper);

    // 按 groupBy 维度分组聚合
    Map<String, List<PerformanceRecordEntity>> groups = new HashMap<>();
    if ("sales_rep".equals(groupBy)) {
        for (PerformanceRecordEntity r : records) {
            groups.computeIfAbsent(String.valueOf(r.getSalesRepId()), ...).add(r);
        }
    } else if ("dept".equals(groupBy)) { /* 按 deptId 分组 */ }
    else if ("zone".equals(groupBy)) { /* 按 zoneId 分组 */ }
    else if ("month".equals(groupBy)) { /* 按 calculateTime 的月份分组 */ }

    // 对每个分组计算：合同总金额、提成总金额、合同数量
    // 通过 Feign 将 ID 转换为可读名称（dimensionName）
    for (Map<String, Object> row : groupedList) {
        if ("sales_rep".equals(groupBy)) {
            Result<?> res = authFeignClient.getUserById(userId);
            row.put("dimensionName", ((Map)res.getData()).get("realName"));
        }
        // ... dept 和 zone 类似，通过 SystemFeignClient 获取名称
    }
    return Map.of("totalContractAmount", ..., "totalCommissionAmount", ...,
                  "groupedList", groupedList);
}
```

##### 业绩排名

**前置依赖**：同业绩汇总。

**实现步骤**：

- **多指标排名查询**：
  1. 前端业绩排名页面提供三种排名指标的切换：按合同金额排名、按提成金额排名、按合同数量排名。
  2. 用户选择排名指标和筛选条件（时间范围、部门、战区等）后，前端发送 GET 请求至 `/api/perfRanking/ranking`。
  3. sales 模块的 `PerfRankingServiceImpl` 根据选择的排名指标，对 `performance_record` 表执行聚合查询，按所选指标降序排列，提取 TOP N 条记录。
  4. 返回排名列表，每条记录包含销售代表姓名、部门名称、战区名称和排名指标的值。
  5. 前端使用 ECharts 横向柱状图或排行榜列表样式渲染排名数据，使管理层能够直观地比较各销售代表的业绩表现。

**核心代码——业绩排名（`PerfRankingServiceImpl.getRankingList()`）**：

```java
public Map<String, Object> getRankingList(String beginTime, String endTime,
        Long deptId, Long zoneId, String rankingType, Integer limit) {
    LambdaQueryWrapper<PerformanceRecordEntity> wrapper = new LambdaQueryWrapper<>();
    wrapper.in(PerformanceRecordEntity::getStatus, 2, 3); // 已确认和已发放
    if (deptId != null) wrapper.eq(PerformanceRecordEntity::getDeptId, deptId);
    if (zoneId != null) wrapper.eq(PerformanceRecordEntity::getZoneId, zoneId);

    List<PerformanceRecordEntity> records = performanceRecordDao.selectList(wrapper);

    // 按 salesRepId 分组聚合：每组合计 contractAmount、commissionAmount、count
    Map<Long, RankingItem> itemMap = new LinkedHashMap<>();
    for (PerformanceRecordEntity r : records) {
        RankingItem item = itemMap.computeIfAbsent(r.getSalesRepId(), RankingItem::new);
        item.addContractAmount(r.getContractAmount());
        item.addCommissionAmount(r.getCommissionAmount());
        item.incrementCount();
    }
    // 填充销售代表姓名（AuthFeignClient）和部门名称（SystemFeignClient）
    for (RankingItem item : itemMap.values()) {
        fillUserName(item);
        fillDeptName(item);
    }
    // 根据 rankingType 排序：contract_amount（默认）、commission、count
    List<RankingItem> sorted = new ArrayList<>(itemMap.values());
    if ("commission".equals(rankingType)) {
        sorted.sort((a, b) -> b.commissionAmount.compareTo(a.commissionAmount));
    } else if ("count".equals(rankingType)) {
        sorted.sort((a, b) -> Integer.compare(b.count, a.count));
    } else {
        sorted.sort((a, b) -> b.contractAmount.compareTo(a.contractAmount));
    }
    // 取前 N 名，设置排名序号
    int rank = 1;
    for (RankingItem item : sorted) {
        if (rank > limit) break;
        rankings.add(Map.of("rank", rank++, "salesRepName", item.salesRepName,
            "contractAmount", item.contractAmount, ...));
    }
    return Map.of("rankings", rankings, "totalSalesReps", sorted.size());
}
```

### 5.2 前端实现

前端项目基于 RuoYi-Vue 3.9.2 脚手架，采用 Vue 3 Composition API（`<script setup>`）+ Element Plus + Pinia + Vue Router + Axios + Vite 技术栈。按业务模块组织在 `src/views/` 目录下，API 调用统一封装在 `src/api/` 目录下。

#### 5.2.1 前端架构概述

**项目目录结构**：

```
src/
  main.js              — 应用入口，注册全局组件/指令/插件
  permission.js         — 路由导航守卫（token 校验、动态路由生成）
  api/                  — API 调用层，按微服务模块分文件
    login.js, menu.js   — 登录、菜单/路由接口
    system/             — 系统管理模块 API（user.js, role.js, dept.js, dict.js...）
    sales/              — 销售模块 API（customer.js, contract.js, contactRecord.js...）
    finance/            — 金融模块 API（bank.js, loanAudit.js, commission.js...）
  store/modules/        — Pinia 状态管理
    user.js             — token、用户信息、角色、权限码
    permission.js       — 动态路由生成（调用 /auth/getRouters）
  router/index.js       — 路由配置（constantRoutes 公开 + dynamicRoutes 动态）
  utils/request.js      — Axios 实例（拦截器、baseURL 动态路由）

  views/                — 页面组件
    system/    (user, role, permission, dept, dict, param, zone)

    finance/   (bank, loan-audit, commission, service-fee, contract-audit)

```

**登录与动态路由流程**：

1. 用户在前端登录页输入用户名密码后点击登录，前端调用 `loginApi(username, password)` 发送 POST 请求至 `/auth/login`。
2. 后端校验通过后返回 token（即用户 ID）。前端将 token 通过 `js-cookie` 写入浏览器 Cookie 并存入 Pinia `userStore.token`。
3. 路由导航守卫 `src/permission.js` 检测到 `userStore.roles` 为空，依次调用：
   - `userStore.getInfo()` → GET `/auth/getInfo`，获取用户 ID、用户名、角色列表（如 `['ROLE_SALES_REP']`）和权限码列表（如 `['system:user:add', '*:*:*']`）。
   - `permissionStore.generateRoutes()` → GET `/auth/getRouters`，后端根据用户的权限码从 `sys_permission` 表中读取出有权限访问的菜单，按树形结构组装后返回。前端 `filterAsyncRouter()` 将每个菜单项解析为 Vue Router 路由对象（`component` 字段由字符串 `"sales/customer/index"` 解析为动态 `import('@/views/sales/customer/index.vue')` 的懒加载组件），通过 `router.addRoute()` 逐个注册。
4. 侧边栏组件 `src/layout/components/Sidebar/index.vue` 读取 `permissionStore.sidebarRouters` 渲染菜单，实现"不同角色看到不同菜单"。

**Axios 请求拦截器——BaseURL 动态路由**：

前端调用后端 API 时，仅发送相对路径（如 `/customer/page`）。`src/utils/request.js` 中的 `getBaseURL()` 函数根据 URL 前缀自动选择代理目标：

```javascript
function getBaseURL(url) {
  if (url.startsWith('/auth/'))       return ''
  if (url.startsWith('/sysUser') || url.startsWith('/sysRole'))
    return '/auth/api'                            // auth 模块
  if (url.startsWith('/sysDict') || url.startsWith('/sysZone'))
    return '/system/api'                          // system 模块
  if (url.startsWith('/customer') || url.startsWith('/contract'))
    return '/sales/api'                           // sales 模块
  if (url.startsWith('/bank') || url.startsWith('/loanAudit'))
    return '/finance/api'                         // finance 模块
  return '/dev-api'                               // 默认回退
}
```

请求拦截器依次执行：① 动态设置 `config.baseURL`；② 注入 `Authorization: Bearer <token>` 请求头；③ 对 POST/PUT 做重复提交检测（sessionStorage 记录最近一次请求的 url+data+time，1 秒内重复提交自动拒绝）。

响应拦截器统一处理：code=200 正常放行；code=401 弹出"登录状态已过期"确认框并跳转登录页；code=500 弹出错误提示。

**权限控制——v-hasPermi 指令**：

`src/directive/permission/hasPermi.js` 注册了全局 Vue 自定义指令 `v-hasPermi`。其在元素挂载（`mounted`）时，从 Pinia `userStore.permissions` 中读取当前用户的权限码列表，与指令参数进行比对。若不匹配，则通过 `el.parentNode.removeChild(el)` 将元素从 DOM 中物理移除。特殊值 `*:*:*` 为超级管理员通配符。

典型用法：
```html
<el-button v-hasPermi="['system:user:add']" @click="handleAdd">新增</el-button>
<el-button v-hasPermi="['system:user:remove']" @click="handleDelete(row)">删除</el-button>
```

此外还有 `v-hasRole`（角色级控制）和 `proxy.$auth.hasPermi()` / `proxy.$auth.hasRole()` 的编程式判断方式。

**通用页面组件模式**：

每个业务模块的列表页组件（`src/views/{module}/{feature}/index.vue`）遵循统一的 Vue SFC 三段式结构：

- **`<template>`**：由上至下依次为搜索表单区（`el-form` :inline + `v-show="showSearch"` 控制显隐）、操作按钮区（`el-row` + `el-button`，按钮带 `v-hasPermi` 指令控制权限）、数据表格（`el-table` + `v-loading` 加载态 + `@sort-change` 排序）、分页器（`Pagination` 全局组件）、新增/编辑对话框（`el-dialog` + `el-form` + `:rules` 校验规则）、详情对话框（`el-dialog` + `el-descriptions` 只读展示）。
- **`<script setup>`**：`import { listXxx, addXxx, updateXxx, delXxx } from '@/api/...'` 导入 API 函数；`const { proxy } = getCurrentInstance()` 获取全局工具代理（`$modal.confirm`、`$modal.msgSuccess` 等）；`const userStore = useUserStore()` 获取登录用户信息用于角色判断。用 `reactive()` 管理表单数据、查询参数和校验规则（通过 `toRefs()` 解构以保持响应性）。核心方法包括 `getList()`（调用 API 分页查询）、`handleAdd()`（重置表单并打开对话框）、`handleUpdate(row)`（通过 API 获取完整数据后预填充表单）、`submitForm()`（表单校验通过后调用 API 写入）、`handleDelete(row)`（弹出确认框后调用逻辑删除）。
- **`<style scoped>`**：组件级局部样式。

#### 5.2.3 销售服务——前端页面构建

销售服务模块的前端页面位于 `src/views/sales/` 目录下。

**客户管理（`sales/customer/index.vue`）**：

这是前端最复杂的列表页面（约 527 行）。核心实现要点：

#### 5.2.5 业绩汇总——前端页面构建

**业绩汇总（`perf-summary/index.vue`）**：

顶部筛选区提供时间范围（`el-date-picker` month 模式）、部门（`el-select`）、战区（`el-select`）、分组维度（`el-select`：按销售人员/部门/战区/月份）。点击查询后调用 `GET /perfSummary/summary`。

**业绩排名（`perf-ranking/index.vue`）**：
