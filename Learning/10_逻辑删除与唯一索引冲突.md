# 逻辑删除与唯一索引冲突问题

> 创建日期：2026-04-24
> 标签：数据库设计 / MySQL / MyBatis-Plus

---

## 一、问题描述

### 1.1 问题现象

使用逻辑删除（软删除）后，出现以下错误：

```
SQLIntegrityConstraintViolationException: Duplicate entry 'liuhongyu' for key 'sys_user.uk_username'
```

### 1.2 问题本质

MySQL 唯一索引默认检查**所有记录**，不考虑 `deleted` 字段：

| id | username | deleted | 唯一索引状态 |
|----|----------|---------|-------------|
| 1001 | admin | 0 | 有效 |
| 1003 | liuhongyu | 0 | 有效 |
| 1003 | liuhongyu | 1 | 软删除后 |

新增 `username='liuhongyu'` 时，虽然 id=1003 的记录已软删除（deleted=1），但唯一索引仍然认为冲突。

---

## 二、解决方案

### 2.1 方案一：唯一索引包含 deleted（推荐）

**原理**：将唯一索引改为联合索引 `(column, deleted)`，使软删除记录不再参与约束。

```sql
-- 原始索引
CREATE UNIQUE INDEX uk_username ON sys_user(username);

-- 修改后
ALTER TABLE sys_user DROP INDEX uk_username;
CREATE UNIQUE INDEX uk_username ON sys_user(username, deleted);
```

**效果**：
| username | deleted | 是否唯一 |
|----------|---------|---------|
| liuhongyu | 0 | ✅ 唯一 |
| liuhongyu | 1 | ✅ 唯一（软删除后是新记录）|

### 2.2 方案二：物理删除

直接 `DELETE FROM table WHERE id = ?`，不留痕迹。

**优点**：简单，不存在唯一索引冲突
**缺点**：数据永久消失，关联数据变成"孤儿"

### 2.3 方案三：禁用账户而非删除

不执行删除操作，而是将 `status = 0`（禁用），保留所有数据。

## 四、额外问题：重复删除报错

### 4.1 问题现象

使用 `uk_username(username, deleted)` 后，新增问题解决，但**同一数据删除两次会报错**：

```sql
-- 第一次删除（UPDATE deleted=1）
UPDATE sys_user SET deleted=1 WHERE id=1003 AND deleted=0;
-- 影响 1 行，成功

-- 第二次删除（再次尝试 UPDATE deleted=1）
UPDATE sys_user SET deleted=1 WHERE id=1003 AND deleted=0;
-- 影响 0 行，MyBatis-Plus 默认抛异常！
```

### 4.2 原因

MyBatis-Plus 默认配置下，逻辑删除找不到记录会抛 `DeleteException`。

### 4.3 解决方案

**方案A**：配置逻辑删除不抛异常
```yaml
mybatis-plus:
  global-config:
    db-config:
      logic-delete-field: deleted
      logic-delete-value: 1
      logic-not-delete-value: 0
      # 关键配置：找不到不抛异常
      # 注意：MyBatis-Plus 3.5.0+ 需要手动处理
```

**方案B**：删除前检查状态
```java
// 在 Service 层先查询
SysUserEntity user = userDao.selectById(id);
if (user != null && user.getDeleted() == 0) {
    userDao.deleteById(id);
}
```

## 五、实际软件中的账户删除策略

### 5.1 核心问题

删除用户/销售代表时，必须考虑以下关联数据：

| 关联表 | 外键字段 | 影响 |
|--------|---------|------|
| contact_record | sales_rep_id | 跟进记录 |
| contract | sales_rep_id | 合同 |
| customer | sales_rep_id | 客户（属于该销售） |
| performance_record | sales_rep_id | 业绩记录 |
| work_log | sales_rep_id | 工作日志 |

### 5.2 常见处理策略

#### 策略一：级联软删除（推荐）

删除用户时，同时软删除/转移其关联数据：

```sql
-- 1. 转移客户到公海
UPDATE customer SET sales_rep_id = NULL, status = 'public_sea' WHERE sales_rep_id = ?;

-- 2. 软删除用户
UPDATE sys_user SET deleted = 1 WHERE id = ?;
```

**优点**：保留历史记录，查询时可显示"已离职"
**缺点**：代码逻辑复杂，可能遗漏关联

#### 策略二：保留原关联，标记已离职（推荐）

不修改关联数据，在显示时做处理：

```javascript
// 前端显示逻辑
{{ scope.row.salesRepName || '已离职' }}
```

**优点**：简单，不破坏数据完整性
**缺点**：销售代表列表仍会显示已离职人员（需过滤）

#### 策略三：物理删除 + 数据迁移

```sql
-- 1. 将该销售的客户转移到其他销售
UPDATE customer SET sales_rep_id = 新销售ID WHERE sales_rep_id = 旧销售ID;

-- 2. 真正删除用户
DELETE FROM sys_user WHERE id = ?;
```

**优点**：数据库干净
**缺点**：操作风险大，需要事务保证

#### 策略四：禁用账户而非删除

```sql
-- 只改变状态，不删除
UPDATE sys_user SET status = 0 WHERE id = ?;
```

**优点**：最安全，可逆
**缺点**：用户列表会越来越长（需要定期归档）

### 5.3 推荐方案

对于本项目，建议采用 **策略二 + 禁用** 的组合：

1. **员工离职**：只修改 `status = 0`（禁用），不执行删除
2. **前端处理**：显示时过滤 `status = 0` 的人员
3. **历史数据**：已软删除的用户显示为"已离职"

```sql
-- 禁用用户（推荐）
UPDATE sys_user SET status = 0 WHERE id = ?;

-- 如果必须删除，至少先禁用一段时间再删除
-- DELETE FROM sys_user WHERE id = ? AND status = 0 AND created_at < DATE_SUB(NOW(), INTERVAL 1 YEAR);
```

### 5.4 业务规则建议

```

│                    员工账号生命周期                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
```
