# 雪花算法 (Snowflake) 详解

> 文档创建时间：2026-04-23
> 关联问题：JavaScript 大数字精度丢失

---

## 一、什么是雪花算法？

**雪花算法 (Snowflake)** 是 Twitter 开源的分布式 ID 生成算法，用于在分布式系统中生成唯一的 ID。

### 核心特点

| 特点 | 说明 |
|------|------|
| **趋势递增** | ID 随时间单调递增，便于数据库索引 |
| **全局唯一** | 分布式环境下不冲突 |
| **高性能** | 本地生成，无需网络调用 |
| **可反解** | ID 中包含时间戳，可获取大致时间 |

---

## 二、ID 结构详解

雪花算法生成的 ID 是一个 **64 位整数**，结构如下：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                           64 位雪花 ID                                    │
├────────────┬────────────────────────────────┬──────────┬──────────────────┤
│  符号位(1) │         时间戳(41位)           │ 机器ID(5)│     序列号(12)    │
│     0      │ 2026-04-22 00:00:00 开始的毫秒数 │ 工作机器ID │ 同一毫秒内序列号  │
└────────────┴────────────────────────────────┴──────────┴──────────────────┘
```

### 各部分详解

| 部分 | 位数 | 说明 | 最大值 |
|------|------|------|--------|
| **符号位** | 1 | 固定为 0，表示正数 | - |
| **时间戳** | 41 | 毫秒级时间戳 | 约 69 年 |
| **机器 ID** | 5 | 数据中心ID(5位) + 机器ID(5位) = 10位 | 1024 个机器 |
| **序列号** | 12 | 每毫秒内的序列号 | 4096 个/毫秒 |

### 时间戳计算

```
41 位时间戳 ≈ 2^41 毫秒 = 2199023255552 毫秒 ≈ 69.7 年

起始时间：2020-01-01 00:00:00（可配置）
最大时间：2090-01-01 00:00:00
```

## 三、为什么会导致 JavaScript 精度丢失？

### 问题根源

```
JavaScript Number 类型使用 64 位浮点数 (IEEE 754)
其中有效整数位数只有 53 位

JavaScript 可安全表示的最大整数：
Number.MAX_SAFE_INTEGER = 2^53 - 1 = 9,007,199,254,740,991

雪花算法 ID：约 64 位整数
当 ID > 9,007,199,254,740,991 时，JavaScript 无法精确表示
```

### 示例演示

```javascript
// 假设雪花 ID
const snowflakeId = 2045710398068801537;

// JavaScript 中的实际值（精度丢失）
console.log(snowflakeId);           // 2045710398068801500
console.log(snowflakeId === 2045710398068801537n) // false

// 精度丢失发生在数字超过 2^53 时
console.log(2**53);        // 9007199254740992 (= MAX_SAFE_INTEGER + 1)
console.log(2**53 + 1);    // 9007199254740992 (加1无效)
console.log(2**53 + 2);    // 9007199254740994 (只能表示偶数)
```

### 精度丢失过程

```
原始 ID:    2045710398068801537
            ↓
JSON.stringify() 或网络传输
            ↓
科学计数法:  2.0457103980688015e+18
            ↓
前端解析:    2045710398068801500  ← 后3位丢失！
```

## 四、本项目的问题分析

### 问题发现

```sql
-- 数据库中的 AUTO_INCREMENT 值
SELECT TABLE_NAME, AUTO_INCREMENT 
FROM information_schema.TABLES 
WHERE TABLE_SCHEMA = 'dafuweng_sales';

-- 结果：
-- contract:    2046843500181372930  ← 雪花 ID
-- customer:     2046879505724682243  ← 雪花 ID
```

### 原因排查

| 项目 | 预期 | 实际 |
|------|------|------|
| `@TableId` 注解 | 指定 IdType.AUTO | 未指定 |
| MyBatis-Plus 配置 | 默认自增 | 可能启用了雪花算法 |
| ID 范围 | 1, 2, 3... | 2046843500181372929... |

### MyBatis-Plus ID 生成策略

```java
// MyBatis-Plus 支持的 ID 类型
public enum IdType {
    AUTO,          // 数据库自增
    INPUT,         // 用户输入
    ASSIGN_ID,     // 雪花算法 (默认)
    ASSIGN_UUID,   // UUID
    NONE           // 无设置
}
```

**注意**：在某些版本或配置下，`@TableId` 不指定时默认使用 `ASSIGN_ID`（雪花算法）

## 五、解决方案

### 方案一：使用自增 ID（推荐）

**修改 common 模块配置**：

```java
// MybatisPlusConfig.java
@Configuration
public class MybatisPlusConfig {
    
    @Bean
    public GlobalConfig globalConfig() {
        GlobalConfig globalConfig = new GlobalConfig();
        GlobalConfig.DbConfig dbConfig = new GlobalConfig.DbConfig();
        dbConfig.setIdType(IdType.AUTO);  // 使用自增 ID
        globalConfig.setDbConfig(dbConfig);
        return globalConfig;
    }
}
```

**优点**：
- ID 小巧（不超过 JavaScript 安全整数范围）
- 顺序性好，利于数据库索引
- 实现简单

**缺点**：
- 分布式环境下需要数据库支持 AUTO_INCREMENT
- 单点写入（高并发场景可能成为瓶颈）

### 方案二：前端使用字符串

**前端修改**：
```javascript
// 不直接使用数字 ID
const idStr = String(row.id)  // 转为字符串
// 或使用后端返回的 idStr 字段
```

**后端修改**：
```java
// 返回时增加字符串形式的 ID
vo.setIdStr(String.valueOf(entity.getId()));

// 接收时用 String，方法内转换
@PostMapping("/{id}")
public Result<Void> update(@PathVariable String id) {
    Long actualId = Long.parseLong(id);
    // ...
}
```

**优点**：
- 无需修改数据库
- 兼容性好

**缺点**：
- 改动较大（涉及所有 API）
- URL 看起来不够简洁

### 方案三：使用分布式 ID 生成器（如 Leaf）

使用美团的 Leaf 等分布式 ID 生成器，支持号段模式，不会产生大数字。

## 八、常见问题

### Q1: 为什么雪花算法生成的 ID 这么大？

雪花算法生成的 ID 本身就是 64 位整数，转换为十进制后会很大。例如：
```
二进制:  00011100100101001000111001100011110011011110110000001100000001
十进制:  2045710398068801537
```

### Q2: 已有的大 ID 数据怎么处理？

1. **保留**：如果业务数据重要，不要删除
2. **迁移**：设计新的自增 ID，通过关联表映射
3. **忽略**：只要新增数据使用自增 ID 即可

### Q3: 什么场景必须用雪花算法？

- 分布式系统，多个服务需要独立生成不冲突的 ID
- 需要按 ID 大致排序（雪花 ID 包含时间戳）
- 高并发写入，避免数据库自增的锁竞争

### Q4: 如何判断当前系统使用的是雪花算法？

```sql
-- 方法1：查看 AUTO_INCREMENT 值
SELECT AUTO_INCREMENT FROM information_schema.TABLES 
WHERE TABLE_SCHEMA = 'your_db' AND TABLE_NAME = 'your_table';
-- 如果值很大（如 > 10^15），可能是雪花算法

-- 方法2：查看 ID 分布
SELECT id FROM your_table ORDER BY id LIMIT 10;
-- 如果从大数字开始且呈递增趋势，可能是雪花算法
```
