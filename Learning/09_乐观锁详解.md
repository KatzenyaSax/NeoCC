# 乐观锁详解

## 一、什么是乐观锁？

### 1.1 概念

**乐观锁（Optimistic Locking）** 是一种并发控制策略，其核心思想是：

> "假设大多数情况下不会发生并发冲突，所以在更新数据时不做锁定，而是在更新时检查数据是否被其他事务修改过。"

### 1.2 对比：乐观锁 vs 悲观锁

| 特性 | 乐观锁 | 悲观锁 |
|------|--------|--------|
| 核心思想 | 假设冲突少，先更新后检查 | 假设冲突多，先锁定再操作 |
| 实现方式 | 版本号/时间戳 | 数据库行锁/表锁 |
| 适用场景 | 读多写少，并发冲突少 | 写多读少，并发冲突多 |
| 性能特点 | 无锁，性能高 | 加锁，可能阻塞 |
| 冲突处理 | 冲突时重试或报错 | 等待锁释放 |

## 二、MyBatis-Plus 中乐观锁的实现

### 2.1 实现方式：版本号（Version）

MyBatis-Plus 使用 `@Version` 注解实现乐观锁：

```java
@Data
@TableName("sys_user")
public class SysUserEntity {
    @TableId(type = IdType.AUTO)
    private Long id;

    @Version  // 标记为版本号字段
    private Integer version;
}
```

### 2.2 工作原理

当执行 `updateById()` 时，MyBatis-Plus 会自动在 SQL 中添加版本条件：

```sql
-- 伪 SQL（实际由 MP 自动生成）
UPDATE sys_user
SET username = ?, version = version + 1, ...
WHERE id = ? AND version = ? AND deleted = 0
```

**执行流程：**

```
┌─────────────────────────────────────────────────────────────┐
│                      更新操作流程                            │
├─────────────────────────────────────────────────────────────┤
│  1. 查询数据 → version = 0                                  │
│       ↓                                                     │
│  2. 应用程序获取 version = 0                                 │
│       ↓                                                     │
│  3. 提交更新 WHERE id = 1 AND version = 0                    │
│       ↓                                                     │
│  4. 数据库执行: version = 0 + 1 = 1                          │
│       ↓                                                     │
│  5. 更新成功，返回 1 行受影响                                │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 冲突检测

如果两个事务同时读取同一条数据：

```
时间线：
T1: 事务A → 读取 user(id=1, version=0) ──────────────────→ 提交更新(where version=0)
T2: 事务B → 读取 user(id=1, version=0) → 提交更新(where version=0)  ← 失败！version已变成1
```

当事务B提交时，数据库发现 `version = 0` 已经不存在了，返回 0 行受影响，MyBatis-Plus 抛出 `OptimisticLockerException`。

## 三、为什么正常情况下不会出现冲突？

### 3.1 单用户操作

在**单用户**场景下：
- 用户A 打开编辑页面 → 读取 version=0
- 用户A 点击保存 → WHERE version=0 匹配 → 更新成功

### 3.2 多用户但有序操作

即使多个用户，只要操作是**串行**的：
- 用户A 保存完成
- 用户B 打开页面（此时 version=1）
- 用户B 保存时 WHERE version=1 匹配 → 更新成功

### 3.3 正常情况汇总

| 场景 | 是否冲突 | 原因 |
|------|----------|------|
| 单用户修改自己的数据 | ❌ | 无并发 |
| 用户A修改，用户B查看 | ❌ | B不修改 |
| A和B按顺序保存 | ❌ | 串行无交叉 |
| 同时打开，同时保存 | ✅ | 并发冲突 |

## 四、为什么会发生这次冲突？

根据日志分析，冲突发生在 **用户管理 → 修改** 功能中。可能的原因：

### 4.1 可能原因1：快速重复点击

```
用户操作：
  点击"修改" → 0.1秒后点击"保存" → 又0.1秒后再次点击"保存"
```

如果第一次请求还未返回，用户再次点击，会发送两次更新请求：
- 请求1: WHERE version=0 → 成功，version变为1
- 请求2: WHERE version=0 → 失败！因为version已是1

### 4.2 可能原因2：表单数据未正确刷新

某些前端框架在更新后未正确刷新表单数据：
```javascript
// 问题代码
function submitForm() {
    updateUser(form.value).then(() => {
        open.value = false
        // 忘记重置表单数据
        // 下次打开时 version 可能不一致
    })
}
```

### 4.3 可能原因3：自动化测试或脚本

可能有定时任务、自动化测试或其他脚本在同时修改数据。

### 4.4 可能原因4：历史数据问题

之前使用雪花ID时创建的用户，version字段可能为NULL或初始值异常：
```sql
-- 检查是否有异常的version值
SELECT id, username, version FROM sys_user WHERE version IS NULL;
```

## 五、如何避免乐观锁冲突？

### 5.1 前端防护

```javascript
// 1. 保存按钮防重复点击
const saving = ref(false)
async function submitForm() {
    if (saving.value) return  // 防止重复提交
    saving.value = true
    try {
        await updateUser(form.value)
    } finally {
        saving.value = false
    }
}

// 2. 更新成功后刷新数据
function submitForm() {
    updateUser(form.value).then(() => {
        proxy.$modal.msgSuccess("修改成功")
        open.value = false
        getList()  // 重新获取列表，确保version最新
    })
}
```

### 5.2 后端优化

#### 方案A：使用 UpdateWrapper（推荐，已实施）

```java
// 不依赖 version 条件，只更新指定字段
var wrapper = new LambdaUpdateWrapper<SysUserEntity>()
        .eq(SysUserEntity::getId, entity.getId())
        .set(SysUserEntity::getRealName, entity.getRealName())
        .set(SysUserEntity::getPhone, entity.getPhone());
sysUserDao.update(null, wrapper);
```

#### 方案B：捕获异常后重试

```java
@Override
public SysUserEntity update(SysUserEntity entity) {
    int retryCount = 0;
    while (retryCount < 3) {
        try {
            sysUserDao.updateById(entity);
            return entity;
        } catch (OptimisticLockException e) {
            retryCount++;
            // 重新获取最新数据后重试
            entity.setVersion(sysUserDao.selectById(entity.getId()).getVersion());
        }
    }
    throw new IllegalStateException("更新失败，请稍后重试");
}
```

#### 方案C：使用悲观锁

如果业务场景确实需要强一致性：

```java
// 使用 SELECT ... FOR UPDATE 悲观锁
public SysUserEntity getUserForUpdate(Long id) {
    return sysUserDao.selectForUpdate(id);
}
```

## 六、总结

### 6.1 乐观锁适用场景

| 场景 | 推荐锁类型 | 原因 |
|------|------------|------|
| 用户资料修改 | 乐观锁 ✅ | 低并发，冲突少 |
| 库存扣减 | 悲观锁 ⚠️ | 高并发，需要强一致 |
| 订单状态流转 | 悲观锁 ✅ | 金额敏感 |
| 点赞数统计 | 乐观锁 ✅ | 最终一致即可 |
| 配置文件修改 | 乐观锁 ✅ | 极少并发 |

### 6.2 本次问题的关键点

```
问题：用户修改 → 乐观锁冲突 → 500错误
原因：updateById() 依赖 version 条件，旧版本号导致更新0行
解决：使用 LambdaUpdateWrapper 只更新必要字段，移除 version 强制校验
```

### 6.3 最佳实践建议

1. **理解业务场景**：选择合适的锁策略
2. **前端防护**：防止重复提交
3. **后端健壮性**：捕获异常并给出友好提示
4. **监控告警**：对乐观锁异常进行监控
5. **数据验证**：更新后验证影响行数

## 七、扩展：其他乐观锁实现方式

### 7.1 时间戳版本

```sql
UPDATE user
SET name = '新名字', update_time = NOW()
WHERE id = 1 AND update_time = '2024-01-01 12:00:00'
```

### 7.2 CAS（Compare-And-Swap）

Java 中的 `AtomicInteger` 使用 CAS：
```java
// 伪代码
while (!atomic.compareAndSwap(oldValue, newValue)) {
    // 失败，重试获取最新值
    oldValue = atomic.get();
}
```
