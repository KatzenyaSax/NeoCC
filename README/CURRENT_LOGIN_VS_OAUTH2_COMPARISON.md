### 1.2 账号密码登录实现

#### 技术架构

```
前端（login.vue）
    ↓ HTTP POST /login
网关（Spring Cloud Gateway）
    ↓ 路由转发
认证服务（Auth Service）
    ↓ 验证用户名密码
数据库（MySQL）
    ↓ 查询用户信息
JWT 工具类
    ↓ 生成 Token
返回 Access Token + Refresh Token
```

#### 核心流程

```java
// 伪代码示例
@PostMapping("/login")
public Result<Map<String, Object>> login(@RequestBody LoginRequest request) {
    // 1. 验证用户名密码
    User user = userService.authenticate(request.getUsername(), request.getPassword());
    
    // 2. 生成 JWT Token
    String accessToken = jwtUtil.generateToken(
        user.getUserId(),
        user.getUsername(),
        user.getRoles(),
        user.getPermissions()
    );
    
    // 3. 生成 Refresh Token
    String refreshToken = tokenStoreService.createRefreshToken(
        user.getUserId(),
        user.getUsername(),
        "web"
    );
    
    // 4. 返回 Token
    return Result.success(Map.of(
        "token", accessToken,
        "refreshToken", refreshToken,
        "expiresIn", 86400
    ));
}
```

#### Token 结构

**Access Token（JWT）**：
```json
{
  "header": {
    "alg": "HS384",
    "typ": "JWT"
  },
  "payload": {
    "jti": "87c5605f-bf1d-4596-a137-262b009396a5",
    "roles": ["common"],
    "perms": [],
    "userId": 1,
    "username": "admin",
    "sub": "1",
    "iat": 1776324742,
    "exp": 1776411142
  }
}
```

**有效期**：
- Access Token：24 小时（86400 秒）
- Refresh Token：7 天（604800 秒）

## 二、OAuth2 协议概述

### 2.1 什么是 OAuth2？

**OAuth2（Open Authorization 2.0）** 是一个开放的授权标准协议，允许第三方应用获取用户在某个服务提供者上的资源访问权限，而无需用户提供用户名和密码。

### 2.2 OAuth2 的核心角色

```
┌──────────┐         ┌──────────┐         ┌──────────┐         ┌──────────┐
│  资源    │         │  资源    │         │  授权    │         │  客户    │
│  所有者  │         │  服务器  │         │  服务器  │         │  端      │
│  (User)  │         │(API)     │         │(Auth)    │         │(Client)  │
└──────────┘         └──────────┘         └──────────┘         └──────────┘
     │                    │                    │                    │
     │  1. 请求授权       │                    │                    │
     │──────────────────────────────────────────────────────────→│
     │                    │                    │                    │
     │  2. 重定向到授权页 │                    │                    │
     │←──────────────────────────────────────────────────────────│
     │                    │                    │                    │
     │  3. 访问授权页面   │                    │                    │
     │───────────────────────────────────────→│                    │
     │                    │                    │                    │
     │  4. 确认授权       │                    │                    │
     │───────────────────────────────────────→│                    │
     │                    │                    │                    │
     │  5. 返回授权码     │                    │                    │
     │←───────────────────────────────────────│                    │
     │                    │                    │                    │
     │  6. 携带授权码     │                    │                    │
     │──────────────────────────────────────────────────────────→│
     │                    │                    │                    │
     │                    │  7. 用授权码换Token │                    │
     │                    │←───────────────────────────────────────│
     │                    │                    │                    │
     │  8. 访问资源       │                    │                    │
     │───────────────────→│                    │                    │
     │                    │                    │                    │
```

| 角色 | 说明 | 示例 |
|------|------|------|
| **资源所有者（Resource Owner）** | 用户本人 | 您 |
| **资源服务器（Resource Server）** | 存储用户资源的服务器 | 微信的 API 服务器 |
| **授权服务器（Authorization Server）** | 颁发 Token 的服务器 | 微信的授权服务器 |
| **客户端（Client）** | 第三方应用 | 您的网站 |

---

### 2.3 OAuth2 的四种授权模式

#### 模式 1：授权码模式（Authorization Code）⭐ 最常用

**适用场景**：有后端服务器的 Web 应用

**流程**：
```
1. 用户点击"用微信登录"
2. 重定向到微信授权页面
3. 用户扫码/确认授权
4. 微信重定向回您的网站，带上 code
5. 您的后端用 code 换取 access_token
6. 登录成功
```

**优点**：
- ✅ 最安全（Token 不经过浏览器）
- ✅ 支持 Refresh Token
- ✅ 适合服务器端应用

---

#### 模式 2：简化模式（Implicit）

**适用场景**：纯前端应用（无后端）

**流程**：
```
1. 用户点击授权
2. 重定向到授权页面
3. 用户确认
4. 直接在 URL 返回 access_token（无 code）
```

**缺点**：
- ❌ 不安全（Token 暴露在 URL 中）
- ❌ 不支持 Refresh Token
- ❌ 已不推荐使用（OAuth 2.1 已废弃）

---

#### 模式 3：密码模式（Resource Owner Password Credentials）

**适用场景**：高度信任的应用（如官方 App）

**流程**：
```
1. 用户输入账号密码
2. 应用直接发送给授权服务器
3. 换取 access_token
```

**缺点**：
- ❌ 需要用户提供密码给第三方
- ❌ 安全性较低
- ❌ 仅限可信应用

---

#### 模式 4：客户端凭证模式（Client Credentials）

**适用场景**：服务器对服务器（无用户参与）

**流程**：
```
1. 应用 A 用 Client ID + Secret
2. 向授权服务器请求 Token
3. 访问应用 B 的 API
```

**特点**：
- ✅ 无需用户参与
- ✅ 适合微服务间调用
- ❌ 不涉及用户授权

## 三、对比分析

### 3.1 账号密码登录 vs OAuth2

| 对比维度 | 当前账号密码登录 | OAuth2 授权码模式 |
|---------|----------------|------------------|
| **认证方式** | 用户名 + 密码 | 第三方授权 |
| **密码存储** | 后端存储（加密） | 不存储用户密码 |
| **安全性** | 依赖密码强度 | 更高（无需给第三方密码） |
| **用户体验** | 需要记住密码 | 一键授权，无需注册 |
| **Token 类型** | JWT（自定义） | JWT/Opaque（标准） |
| **权限控制** | 角色 + 权限 | Scope（权限范围） |
| **Token 刷新** | Refresh Token | Refresh Token |
| **标准协议** | ❌ 自定义 | ✅ OAuth2 标准 |
| **第三方集成** | ❌ 不支持 | ✅ 支持 |
| **适用场景** | 自有用户体系 | 借用第三方用户体系 |

#### 详细对比

**1. 认证流程**

```
【当前账号密码登录】
用户输入账号密码 → 后端验证 → 生成 JWT → 返回前端

【OAuth2 授权码模式】
用户点击授权 → 重定向到第三方 → 用户确认 → 获取 code → 后端换 token → 返回前端
```

**2. 安全性对比**

| 安全特性 | 当前实现 | OAuth2 |
|---------|---------|--------|
| 密码泄露风险 | 高（用户可能重复使用密码） | 低（不暴露密码） |
| Token 被盗 | 中等（JWT 可能被窃取） | 低（短期 Token + 刷新机制） |
| CSRF 防护 | 需自行实现 | state 参数标准防护 |
| 权限控制 | 粗粒度（角色） | 细粒度（Scope） |
| 授权撤销 | 需手动实现 | 标准撤销端点 |

**3. 代码复杂度**

```
【当前实现】
- 简单直接
- 代码量少
- 易于理解和维护
- 约 50 行核心代码

【OAuth2 实现】
- 流程复杂
- 需要处理重定向、code 交换等
- 需要遵循标准规范
- 约 200+ 行核心代码
```

### 3.2 第一方扫码登录 vs OAuth2

| 对比维度 | 当前扫码登录 | OAuth2 授权码模式 |
|---------|-------------|------------------|
| **交互方式** | 扫描二维码 | 重定向授权 |
| **协议标准** | ❌ 自定义协议 | ✅ OAuth2 标准 |
| **授权流程** | 轮询状态 | 回调机制 |
| **实时性** | 低（2秒轮询间隔） | 高（即时回调） |
| **Token 获取** | 轮询查询状态 | code 换取 |
| **设备要求** | 需要两个设备/窗口 | 单个设备即可 |
| **第三方集成** | ❌ 仅自有系统 | ✅ 可集成第三方 |
| **适用场景** | 内部系统、演示 | 开放平台、第三方登录 |
| **实现难度** | 中等 | 较高 |

#### 详细对比

**1. 架构对比**

```
【当前扫码登录】
Web 端生成二维码 → 手机扫码 → 管理后台确认 → Web 端轮询获取 Token

【OAuth2 扫码登录（如微信）】
Web 端显示微信二维码 → 微信扫码 → 微信服务器回调 → 后端获取 Token
```

**2. 实时性对比**

| 机制 | 延迟 | 资源消耗 |
|------|------|---------|
| 轮询（当前） | 0-2 秒 | 高（频繁请求） |
| 回调（OAuth2） | < 1 秒 | 低（按需请求） |
| WebSocket（可选） | < 100ms | 中等（长连接） |

**3. 状态管理对比**

```
【当前实现】
内存存储（ConcurrentHashMap）
- 优点：快速、简单
- 缺点：服务重启丢失、无法水平扩展

【OAuth2 标准实现】
数据库/Redis 存储授权码
- 优点：持久化、可分布式
- 缺点：需要额外存储
```

### 3.3 核心差异总结

#### 相同点

| 特性 | 说明 |
|------|------|
| **最终目标** | 都是实现用户身份认证 |
| **Token 机制** | 都使用 Token 进行后续请求认证 |

| **社区支持** | 无 | 强大 |
| **审计合规** | 需自行实现 | 符合国际标准 |
| **学习曲线** | 低 | 中等 |

## 九、总结

### 9.1 核心结论

1. **当前实现完全满足课程设计需求**
   - 功能完整
   - 代码简洁
   - 演示效果好

2. **OAuth2 是国际标准，但不是必须的**
   - 适合开放平台和第三方集成
   - 对于封闭系统可能过度设计

3. **两种方式可以共存**
   - 当前实现作为基础
   - 后续可扩展为 OAuth2

4. **扫码登录和 OAuth2 不是对立的**
   - 扫码登录是交互方式
   - OAuth2 是底层协议
   - 可以用 OAuth2 实现扫码登录
